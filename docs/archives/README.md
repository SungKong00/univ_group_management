# 📚 아카이브 (Archives)

중요한 개선 작업, 트러블슈팅, 기술적 의사결정을 기록합니다.

## 디렉토리 구조

```
docs/archives/
├── performance/         (성능 최적화)
├── architecture/        (아키텍처 결정)
├── troubleshooting/     (버그 해결)
└── migrations/          (시스템 마이그레이션)
```

## 파일명 규칙

```
YYYY-MM-DD-구체적인-문제-설명.md
```

**좋은 예시**:
- `2025-10-24-member-query-n1-problem.md`
- `2025-10-25-h2-with-recursive-parsing-error.md`

**나쁜 예시**:
- `optimization.md` (날짜 없음, 불명확)
- `2025-10-24-fix.md` (무엇을 고쳤는지 불명확)

## 문서 구조 (필수)

```markdown
# [제목: 문제를 명확히 표현]

**날짜**: YYYY-MM-DD
**관련 커밋**: [커밋 해시]

## 문제
무엇이 문제였는가? (증상, 성능 지표, 에러 로그)

## 문제 원인
왜 이 문제가 발생했는가? (근본 원인, 기술적 배경)

## 해결방안 분석
어떤 접근법들을 고려했는가?

## 해결방안 후보
1. **방안 A**: 설명, 장단점
2. **방안 B**: 설명, 장단점
3. **방안 C**: 설명, 장단점

## 해결방안 선택 이유
왜 특정 방안을 선택했는가? (성능, 유지보수성, 호환성 등)

## 개선 후 개선점
- 정량적 개선: 수치로 측정 가능한 개선
- 정성적 개선: 코드 품질, 유지보수성 등
```

## 언제 아카이브?

- 성능 최적화 (N+1, 인덱스, 캐싱)
- 3시간 이상 소요된 버그 해결
- 아키텍처 변경 결정
- 복잡한 마이그레이션

## 100줄 원칙

- 핵심만 간결하게
- 상세 코드는 커밋 링크로 대체
- 필요시 여러 문서로 분리
