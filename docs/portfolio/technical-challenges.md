# 기술적 도전과 해결

4개월 동안 개발하며 실제로 직면했던 문제들입니다. 백엔드에서 익숙했던 문제도 있었으나, 프론트엔드를 처음 경험하며 마주한 생소한 문제들도 많았습니다. 각각의 문제를 어떻게 해결했는지, 그리고 그 과정에서 무엇을 배웠는지 정리했습니다.

---

## 1. 권한 확인 시 DB 과부하 문제 (N+1 문제)

### 문제 상황

권한 확인에 필요한 정보가 여러 테이블에 분산되어 있었습니다. 사용자 → 그룹 멤버십 → 역할 → 채널 바인딩을 순서대로 조회해야 했기에, 채널이 10개인 그룹에서 하나의 권한을 확인하는 데 총 12번의 쿼리가 실행되는 문제가 발생했습니다.

JPA 수업을 통해 N+1 문제를 이론적으로는 알고 있었으나, 실제 프로젝트 코드에서 이를 마주하는 것은 또 다른 경험이었습니다. `@OneToMany` 관계를 적절한 고려 없이 사용하고 있었으며, Hibernate SQL 로그를 확인한 결과 수많은 쿼리가 반복적으로 실행되고 있음을 확인했습니다.

### 해결 방법

**Fetch Join과 2단계 쿼리**를 활용하여 DB 조회 횟수를 항상 2회 이하로 대폭 단축했습니다.

1단계에서 역할에 연결된 채널 ID를 한 번에 조회하고, 2단계에서 해당 ID들을 사용하여 채널 정보를 벌크 조회(Bulk Read)하는 방식을 채택했습니다. JPQL의 `JOIN FETCH`를 처음 실무에 적용해 보며, 한 번의 쿼리로 연관 엔티티를 모두 로딩하는 효율성을 직접 체감할 수 있었습니다.

또한 Caffeine 캐시를 도입하여 동일한 사용자가 같은 채널에 접근할 경우 DB 조회를 생략하도록 개선했습니다. Spring의 `@Cacheable` 어노테이션을 통해 적용 자체는 용이했으나, 데이터 정합성을 위한 캐시 무효화 시점을 설정하는 과정이 까다로웠습니다. 역할 변경, 채널 바인딩 수정, 멤버 탈퇴 등 권한에 영향을 주는 모든 시점을 파악하여 적절히 캐시를 갱신하도록 구현했습니다.

이 과정을 통해 권한 확인처럼 모든 API에서 공통적으로 발생하는 로직의 성능이 시스템 전체의 응답 속도에 얼마나 큰 영향을 미치는지 깊이 이해하게 되었습니다.

## 2. 세밀한 권한 제어를 위한 하이브리드 모델 설계

### 문제 상황

초기에는 Spring Security의 기본 기능처럼 간단한 역할 기반 제어(RBAC)로 충분할 것이라 예상했습니다. 그러나 개발을 진행하며 각 채널의 성격에 따라 서로 다른 권한이 필요하다는 점을 인지하게 되었습니다.

```
공지사항 채널: 멤버 → 읽기 전용
자유게시판:    멤버 → 읽기 및 쓰기 가능
비공개 채널:   멤버 → 접근 차단 (바인딩이 없는 경우)
```

단순한 시스템 역할만으로는 이러한 세밀한 차이를 표현하기 어려웠습니다. Spring Security의 `@PreAuthorize("hasRole('ADMIN')")`와 같은 방식으로는 "특정 채널에서의 특정 역할의 행위"를 정의하는 데 한계가 있었습니다.

### 해결 방법

**RBAC와 Channel Override를 결합한 하이브리드 모델**로 전환했습니다.

그룹 수준에서 시스템 역할(그룹장/교수/멤버)이 기본적인 권한을 정의하고, 채널 수준에서 `ChannelRoleBinding`을 통해 역할별 권한을 오버라이드할 수 있도록 설계했습니다.

이때 가장 중요하게 고려한 원칙은 **Secure by Default**입니다. 새로 생성된 채널은 어떠한 바인딩도 존재하지 않으므로 명시적인 권한 부여 전까지 접근이 불가능합니다. 이는 Spring Security의 "모든 요청 거부" 정책과 궤를 같이하며, 의도치 않은 정보 노출을 원천적으로 차단합니다.

또한 시스템 역할 3종은 삭제 및 수정이 불가능한 불변 상태로 관리하도록 구현했습니다. 그룹장과 같은 핵심 역할이 삭제될 경우 시스템의 무결성이 훼손될 수 있기 때문입니다. 이를 위해 도메인 엔티티에 `isSystemRole` 플래그를 추가하고, 서비스 레이어에서 관련 요청 시 예외를 발생시키도록 처리했습니다.

## 3. Navigator 2.0 도입을 통한 선언적 라우팅 구현

### 문제 상황

Spring에서의 라우팅은 `@GetMapping` 등 어노테이션을 통해 특정 URL 요청을 처리하는 비교적 명확한 구조를 가집니다. 하지만 프론트엔드 라우팅은 이와 상당히 달랐습니다.

Flutter의 기본 Navigator는 스택 기반의 명령형(Imperative) 방식입니다. 화면을 쌓고 제거하는 구조적 특성상, 그룹 전환이나 딥 링크 처리, 권한 기반의 페이지 폴백(Fallback)이 필요한 상황에서 상태 관리의 복잡도가 급격히 증가했습니다.

주요 이슈는 다음과 같았습니다:
- 그룹 전환 시 뒤로가기 버튼 동작이 예상과 다르게 작동하는 문제
- URL을 통한 직접 접근 시 화면 스택이 올바르게 형성되지 않는 문제
- 권한이 없는 페이지 접근 시의 자동 이동 로직이 불안정하게 동작하는 문제

백엔드에서는 URL이 곧 리소스의 상태를 의미하지만, 프론트엔드에서는 URL, 화면 스택, 권한 정보, 이전 상태가 유기적으로 맞물려야 함을 깨달았습니다.

### 해결 방법

Navigator 2.0을 도입하여 선언적(Declarative) 라우팅으로 전면 개편했습니다. 총 7단계에 걸친 점진적 전환을 진행했으며, 각 단계마다 철저한 검증 테스트를 병행했습니다.

특히 **Context-Aware Group Switching** 구현에 공을 들였습니다. 그룹 전환 시 권한을 즉시 확인하고, 권한이 없는 경우 최근 접근했던 그룹으로 안전하게 폴백하도록 설계했습니다. 명령형 방식에서는 복잡한 조건문과 화면 조작 로직이 필요했으나, 선언적 방식에서는 "상태의 변화가 UI의 변화로 이어지는" 구조를 통해 훨씬 간결하게 표현되었습니다.

이 과정을 통해 "선언적" 프로그래밍의 핵심 가치를 이해하게 되었습니다. 현재 상태에 따른 최종 화면의 모습을 정의해두면, 상태 변경에 따라 UI가 자동으로 동기화되므로 조건 분기 로직이 획기적으로 줄어들었습니다. 또한 LRU 캐시를 활용한 스크롤 위치 보존, 디바운싱을 통한 중복 클릭 방지, 네트워크 에러 자동 재시도 로직 등을 추가하여 안정성을 높였습니다.

그 결과, 총 98개의 라우팅 관련 테스트를 모두 통과하며 견고한 네비게이션 시스템을 구축할 수 있었습니다.

## 4. Post ID 기반의 정확한 읽음 상태 추적

### 문제 상황

사용자가 채널에 다시 들어왔을 때 "새 글 알림"과 같은 기능을 제공하기 위해 안 읽은 글을 정확히 식별해야 했습니다. 초기에는 "마지막 방문 시간" 타임스탬프를 서버에 저장하고 이를 기준으로 이후 작성된 글을 판단하는 방식을 사용했습니다.

하지만 서버와 클라이언트 간의 미세한 시간 차이, 데이터 생성 시점과 화면 렌더링 시점의 괴리로 인해 방금 읽은 글이 여전히 안 읽은 상태로 표시되는 등의 정합성 문제가 간헐적으로 발생했습니다.

### 해결 방법

**시간 기반에서 Post ID 기반 방식**으로 전환했습니다. "마지막으로 읽은 게시글의 고유 ID"를 서버에 기록하고, 현재 로드된 리스트에서 해당 ID보다 큰 값을 가진 게시글을 미읽음으로 판별합니다. 자동 증가하는 ID 값을 활용함으로써 순서 보장이 명확해졌고 시간 동기화에 따른 변수도 제거되었습니다.

구현 중에는 **Race Condition** 이슈도 경험했습니다. Flutter의 `VisibilityDetector` 위젯이 해제된 이후에도 예약된 비동기 콜백이 실행되어 이미 해제된 Provider에 접근하려는 에러가 있었습니다.

Spring의 `@Async` 방식과는 다른 Flutter의 단일 스레드 이벤트 루프 특성을 이해하는 계기가 되었습니다. 위젯의 생명주기가 끝난 뒤에도 이벤트 큐에 남은 콜백이 실행될 수 있음을 확인하고, `mounted` 체크 로직을 추가하여 위젯이 활성 상태일 때만 콜백이 동작하도록 수정하여 해결했습니다.

## 5. Optimistic UI 적용을 통한 사용자 경험 개선

### 문제 상황

댓글 작성 시 서버 응답까지 발생하는 1~2초의 지연 시간이 사용자에게는 서비스가 멈춘 듯한 부정적인 경험을 줄 수 있었습니다.

백엔드 관점에서는 요청 처리 후 응답을 보내는 것이 정석적인 흐름이지만, 프론트엔드에서는 이러한 짧은 대기 시간조차 체감 성능에 큰 영향을 미칩니다. 사용자 인터랙션에 즉각적으로 반응하지 않는 UI는 서비스의 품질을 낮추는 요인이 되었습니다.

### 해결 방법

**Optimistic Update(낙관적 업데이트)** 패턴을 도입했습니다. 사용자가 작성 버튼을 누르는 즉시 화면에 댓글을 임시로 표시하고, 백그라운드에서 서버 요청을 처리합니다. 요청이 성공하면 서버에서 생성된 실제 데이터로 정보를 업데이트하고, 만약 실패할 경우에는 화면에서 해당 항목을 제거하고 롤백한 뒤 에러 메시지를 표시합니다.

백엔드 개발자로서는 데이터 확정 전 UI를 갱신하는 것이 생소했으나, 대다수의 요청이 성공하는 일반적인 상황에서 사용자 경험을 최우선으로 고려하는 프론트엔드 특유의 사고방식을 이해하게 되었습니다. 이를 Riverpod의 `AsyncNotifier`와 결합하여 상태 변화의 흐름을 체계적으로 관리했으며, 다양한 시나리오에 대한 테스트를 통해 안정성을 검증했습니다.

## 6. Modular Monolith 및 Clean Architecture 도입

### 문제 상황

프로젝트 규모가 커지면서 기존의 단순 모놀리식 구조가 한계에 봉착했습니다. 17개의 컨트롤러가 하나의 패키지에 집중되어 있었고, 서비스 간의 직접적인 상호 참조로 인해 특정 기능을 수정할 때마다 의도치 않은 사이드 이펙트가 발생하는 스파게티 코드가 되어가고 있었습니다.

기술 계층(Controller, Service, Repository) 중심의 전형적인 패키지 구조는 도메인이 복잡해질수록 관련 없는 코드들이 섞이는 문제를 야기했습니다.

### 해결 방법

`backend_new/` 디렉토리를 통해 6개 핵심 도메인으로 분리된 Clean Architecture를 새롭게 구축했습니다.

계층 중심이 아닌 **도메인 중심(User, Group, Permission, Workspace, Content, Calendar)으로 패키지를 재구성**했습니다. 각 도메인은 독립적인 계층 구조를 가지며, 도메인 간 통신은 반드시 Service 인터페이스를 통해서만 이루어지도록 강제하여 결합도를 낮췄습니다.

총 7단계의 페이즈로 나누어 체계적인 개편을 진행했으며, 155개의 테스트 코드를 작성하여 아키텍처 변경에 따른 안정성을 확보했습니다. 또한 실제 운영 환경으로의 전환을 위해 '읽기 우선 전환 → Feature Flag 기반 쓰기 전환 → 완전 전환'이라는 3단계 마이그레이션 전략을 수립하여 리스크를 최소화하고자 노력했습니다.

## 7. 디자인 시스템 구축을 통한 UI 일관성 확보

### 문제 상황

기능 구현에 집중하다 보니 UI의 일관성이 무너지는 문제가 발생했습니다. 동일한 성격의 버튼이나 입력 필드가 페이지마다 조금씩 다른 형태를 띠게 되었고, 색상값이나 여백 수치들이 코드 곳곳에 하드코딩되어 유지보수가 매우 어려워졌습니다.

백엔드에서 응답 형식을 통일하여 일관성을 유지하는 것과 달리, UI의 일관성은 훨씬 더 세밀한 규칙과 관리가 필요함을 깨달았습니다.

### 해결 방법

두 가지 전략을 병행하여 해결을 시도했습니다.

첫째, 기존 코드에 디자인 토큰을 도입했습니다. 하드코딩된 수치들을 `AppSpacing`, `AppColors` 등의 정의된 토큰으로 대체하고, 반복되는 UI 패턴을 `AppFormField`, `DialogHelpers` 등의 공통 컴포넌트로 추출하여 코드 양을 줄이고 재사용성을 높였습니다.

둘째, `frontend_new/`에서 Bottom-up 방식의 디자인 시스템 구축을 실험했습니다. 40개의 시맨틱 컬러와 39개의 핵심 컴포넌트를 미리 정의하고 이를 조합하여 화면을 구성하는 방식을 시도했습니다. 컴포넌트 쇼케이스를 통해 디자인 일관성을 시각적으로 확인하며 개발 효율이 향상됨을 확인했습니다.

이 과정을 통해 **디자인 시스템은 개발 후반부의 수정 사항이 아니라, 프로젝트 초기 단계에서 반드시 확립해야 하는 기초 설계**임을 깊이 인식하게 되었습니다.

## 8. 프론트엔드 상태 관리와 의존성 설계

### 문제 상황

Stateless한 특성을 가진 백엔드와 달리, 프론트엔드는 인증 상태, 선택된 그룹 정보 등 다양한 전역 상태가 메모리상에 유지되어야 합니다. 초기 설계 시 이러한 상태들 사이의 의존 관계가 복잡하게 얽히면서, 특정 상태 변경 시 연쇄적인 갱신이 제어되지 않거나 로그아웃 시 순환 참조로 인해 상태 초기화가 실패하는 등의 심각한 버그가 발생했습니다.

### 해결 방법

Provider 간의 **의존성 방향을 단방향**으로 엄격히 제한했습니다.

`auth → user → groups → selectedGroup → workspace → channels → selectedChannel → posts`로 이어지는 흐름을 정의하고, 이를 역행하는 참조를 차단했습니다. 인증 로직을 별도의 Repository로 분리하여 순환 참조의 고리를 끊어냈습니다.

이는 Spring의 서비스 간 순환 참조 방지 원리와도 일맥상통합니다. 의존성 방향을 명확히 하고 공통 로직을 분리함으로써 시스템의 예측 가능성을 높였습니다. `AsyncNotifier`를 통해 로딩 및 에러 상태를 통합 관리하고, 상위 상태 변화에 따라 하위 상태가 유기적으로 갱신되는 반응형 구조를 확립하여 안정적인 애플리케이션 상태를 유지할 수 있게 되었습니다.
