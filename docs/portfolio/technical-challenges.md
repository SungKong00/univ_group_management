# 기술적 도전과 해결

4개월 동안 개발하면서 실제로 부딪힌 문제들이다. 백엔드에서 익숙했던 문제도 있고, 프론트엔드를 처음 하면서 만난 생소한 문제도 있다. 각각 어떻게 풀었는지, 그리고 그 과정에서 뭘 배웠는지 정리했다.

---

## 1. 권한 확인할 때마다 DB를 12번 찌르고 있었다

### 문제

권한 확인에 필요한 정보가 여러 테이블에 흩어져 있었다. 사용자 → 그룹 멤버십 → 역할 → 채널 바인딩을 순서대로 조회해야 해서, 채널이 10개인 그룹에서 하나의 권한을 확인하는 데 12번의 쿼리가 나갔다.

JPA 수업에서 N+1 문제를 배우긴 했는데, 교과서에서 보는 것과 내 코드에서 실제로 만나는 건 다른 느낌이었다. `@OneToMany`를 아무 생각 없이 쓰고 있었는데, Hibernate가 뱉는 SQL 로그를 켜보니 쿼리가 쏟아지고 있었다.

### 해결

**Fetch Join + 2단계 쿼리**로 항상 2번 이하의 DB 조회로 줄였다.

1단계에서 역할에 연결된 채널 ID를 한 번에 가져오고, 2단계에서 그 ID들로 채널 정보를 벌크 조회한다. JPQL의 `JOIN FETCH`를 처음 써봤는데, 한 방 쿼리로 연관 엔티티를 다 가져오는 게 신기했다.

여기에 Caffeine 캐시를 추가했다. 같은 사용자가 같은 채널에 접근할 때는 DB를 아예 안 찌른다. Spring에서 `@Cacheable` 한 줄이면 되니까 적용 자체는 어렵지 않았는데, 캐시 무효화 시점을 정하는 게 생각보다 까다로웠다. 권한이 바뀔 때만 캐시를 날려야 하는데, "권한이 바뀌는 시점"이 역할 변경, 채널 바인딩 수정, 멤버 탈퇴 등 여러 군데에 걸쳐 있었기 때문이다.

이 작업을 하면서 "권한 확인은 모든 API에서 일어나니까, 여기가 느리면 전체가 느려진다"는 걸 체감했다.

## 2. 단순 RBAC로는 안 되겠다

### 문제

처음에는 Spring Security처럼 간단하게 될 줄 알았다. `if (role == "ADMIN")` 같은 하드코딩으로 시작했는데, 채널마다 다른 권한이 필요하다는 걸 금방 깨달았다.

```
공지사항 채널: 멤버 → 읽기만
자유게시판:    멤버 → 읽기 + 쓰기
비공개 채널:   멤버 → 접근 자체가 안 됨
```

역할 하나로는 이런 차이를 표현할 수 없었다. Spring Security의 `@PreAuthorize("hasRole('ADMIN')")` 같은 단순 체크로는 "이 채널에서 이 역할은 뭘 할 수 있나"를 표현하지 못했다.

### 해결

**RBAC + Channel Override 하이브리드** 모델로 전환했다.

그룹 수준에서 시스템 역할(그룹장/교수/멤버)이 기본 권한을 정하고, 채널 수준에서 `ChannelRoleBinding`으로 역할별 권한을 오버라이드한다.

이때 가장 신경 쓴 원칙은 **Secure by Default**다. 새로 만든 채널은 아무 바인딩도 없어서, 명시적으로 열어줘야만 접근된다. Spring Security의 기본 정책이 "모든 요청 거부"인 것과 같은 발상이다. "실수로 열려있는" 상황을 원천 차단한다.

시스템 역할 3종(그룹장/교수/멤버)은 삭제/수정 불가(불변)로 만들었다. 누군가 실수로 그룹장 역할을 지워버리면 시스템이 통째로 깨지기 때문이다. 이건 도메인 엔티티에 `isSystemRole` 플래그를 두고, 서비스 레이어에서 삭제/수정 요청이 오면 예외를 던지는 식으로 구현했다.

## 3. Navigator 2.0 — 프론트엔드 라우팅이 이렇게 복잡한 줄 몰랐다

### 문제

Spring에서 라우팅은 간단하다. `@GetMapping("/groups/{id}")` 붙이면 해당 URL로 요청이 온다. 끝이다. 그래서 프론트엔드 라우팅도 비슷할 거라고 생각했는데, 전혀 달랐다.

Flutter의 기본 Navigator는 push/pop 기반의 명령형이다. 화면을 쌓고 빼는 스택 구조인데, 이 프로젝트처럼 그룹 전환, 딥 링크, 권한 기반 폴백이 필요한 경우에는 상태가 계속 꼬였다.

구체적으로 이런 문제들이 있었다:

- 그룹 A에서 그룹 B로 전환했는데, 뒤로가기를 누르면 엉뚱한 화면이 나옴
- URL로 특정 채널에 직접 접근하면 중간 화면이 빠져서 뒤로가기가 안 됨
- 권한 없는 그룹으로 이동할 때 기본 그룹으로 자동 이동해야 하는데, 명령형으로는 이게 안정적이지 않음

Spring에서는 URL이 곧 상태다. `/groups/3/channels/5` 요청이 오면 서버가 해당 리소스를 찾아서 응답하면 끝이다. 그런데 프론트엔드에서는 URL, 화면 스택, 사용자 권한, 이전 상태가 전부 맞물려야 한다. 이 복잡도가 처음에는 이해가 안 됐다.

### 해결

Navigator 2.0(선언적 라우팅)으로 전면 전환했다. 7단계로 나눠서 점진적으로 진행했고, 각 단계마다 테스트를 추가했다.

가장 까다로웠던 건 **Context-Aware Group Switching**이었다. 그룹을 전환할 때 권한을 확인하고, 권한이 없으면 가장 최근에 접근했던 다른 그룹으로 폴백한다. 명령형에서는 "if 권한없음 → pop → push 다른그룹"을 직접 짜야 했는데, 선언적 방식에서는 "상태가 바뀌면 UI가 알아서 따라온다"로 깔끔하게 표현됐다.

이 "선언적"이라는 개념을 이해하는 데 시간이 좀 걸렸다. Spring MVC에서 "이 요청이 오면 이렇게 처리해라"고 하나씩 지정하는 게 명령형이고, 선언적은 "현재 상태가 이러면 화면은 이래야 한다"고 결과를 선언하는 거다. 상태만 바꾸면 UI가 알아서 맞춰지니까, 조건 분기가 훨씬 줄었다.

스크롤 위치 보존은 LRU 캐시(최근 5개 채널)로 구현했다. 빠른 연속 클릭은 디바운싱으로, 네트워크 에러는 자동 재시도로 처리했다.

**결과**: 98개 테스트 (단위 30 + 위젯/통합 58 + 성능/접근성 10) 전부 통과.

## 4. 읽지 않은 글 추적 — 시간 기반의 함정

### 문제

채널을 벗어났다 돌아왔을 때, "새 글 3개"처럼 안 읽은 글 수를 보여줘야 했다. 처음에 "마지막 방문 시간"을 기준으로 만들었다. 서버에 `lastVisitedAt` 타임스탬프를 저장하고, 그 이후에 작성된 글을 세는 방식이었다.

문제는 시간이 생각만큼 정확하지 않다는 거였다. 서버 시간과 클라이언트 시간이 미세하게 다르고, 글 작성 시간과 실제 화면에 표시되는 시간 사이에도 차이가 있었다. 결과적으로 "방금 읽은 글이 미읽음으로 뜨는" 버그가 간헐적으로 발생했다.

### 해결

**시간 기반 → Post ID 기반**으로 전환했다. "마지막으로 읽은 게시글 ID"를 서버에 저장하고, 그보다 큰 ID의 게시글을 "미읽음"으로 처리한다. ID는 자동 증가(auto-increment)니까 순서가 보장되고, 시간 동기화 문제도 없다.

구현 과정에서 예상 못한 **Race Condition**을 만났다. Flutter의 `VisibilityDetector` 위젯이 화면에서 사라진(dispose된) 후에도, 이미 예약된 비동기 콜백이 실행되면서 해제된 Provider ref에 접근하는 에러가 발생했다.

Spring에서 비동기를 쓸 때는 `@Async`를 붙이고 스레드 풀을 설정하면 되지만, Flutter의 비동기는 다르다. 단일 스레드에서 이벤트 루프로 동작하는데, 위젯이 사라져도 이미 큐에 들어간 콜백은 실행된다. 이걸 몰라서 한참 삽질했다. 결국 `mounted` 체크를 추가해서 "위젯이 아직 살아있을 때만 콜백을 실행"하도록 수정해서 해결했다.

## 5. 서버 응답을 안 기다린다고?

### 문제

댓글을 작성하면 서버 응답까지 1-2초가 걸렸다. 그동안 화면이 멈춘 것처럼 보이는 게 신경 쓰였다.

백엔드 개발자 입장에서는 "서버에 요청 보내고 → 응답 받고 → 결과 보여주기"가 당연한 흐름이다. Spring에서 `@PostMapping`으로 댓글을 생성하면, 클라이언트는 응답이 올 때까지 기다린다. 그런데 프론트엔드에서는 사용자가 그 1-2초를 체감한다. 화면이 멈춘 것 같은 느낌을 주면 사용자 경험이 나빠진다.

### 해결

**Optimistic Update(낙관적 업데이트)** 패턴을 적용했다. 작성 버튼을 누르면 즉시 화면에 댓글을 추가하고, 서버 응답이 오면 실제 데이터(서버가 부여한 ID, 생성 시간 등)로 교체한다. 서버가 에러를 반환하면 화면에서 댓글을 제거(롤백)한다.

```
1. 사용자가 댓글 작성 → 즉시 화면에 표시 (임시 데이터)
2. 서버에 POST 요청 전송
3-a. 성공 → 서버 데이터로 교체
3-b. 실패 → 화면에서 제거 + 에러 메시지
```

처음에 이 패턴을 알았을 때 "서버 확인도 안 하고 화면에 보여준다고?" 싶었다. 백엔드에서는 데이터 정합성이 최우선인데, 프론트엔드에서는 사용자 경험이 곧 기능이다. 대부분의 요청은 성공하니까 일단 보여주고, 드물게 실패하면 그때 롤백한다는 발상이 백엔드 사고방식과 많이 달랐다.

이 패턴을 Riverpod의 `AsyncNotifier`에 적용해서, 상태 변경(낙관적 추가 → 서버 확인 → 교체/롤백) 흐름을 한 곳에서 관리한다. 29개 테스트로 성공/실패/네트워크 에러 시나리오를 검증했다.

## 6. 모놀리식이 한계에 부딪혔다

### 문제

이건 백엔드니까 오히려 익숙한 문제였다. 기능이 늘면서 17개 컨트롤러가 한 패키지에 들어가 있는 모놀리식 구조의 한계가 왔다. 서비스끼리 직접 참조하고 있어서, `CalendarService`를 수정하면 `GroupService`가 깨지고, `GroupService`를 수정하면 `WorkspaceService`가 깨지는 식이었다.

Spring에서 패키지 구조를 어떻게 잡을지 고민해본 적이 없었다. `controller`, `service`, `repository`, `entity`로 기술 계층별로 나누는 게 기본이라고만 배웠는데, 기능이 6개 도메인으로 늘어나니까 한 패키지에 관련 없는 코드가 뒤섞이는 문제가 생겼다.

### 해결

`backend_new/`에서 6개 도메인으로 분리한 Clean Architecture를 처음부터 다시 구축했다.

기술 계층별이 아니라 **도메인별로 패키지를 나눴다**: User, Group, Permission, Workspace, Content, Calendar. 각 도메인 안에서 entity, service, repository, controller를 갖는 구조다. 도메인 간에는 반드시 Service 인터페이스를 통해서만 통신하도록 강제했다.

7단계(Phase 0~7)로 나눠서 진행했고, 155개 테스트를 작성해서 구조 변경의 안전망을 만들었다.

기존 백엔드와의 전환을 위해 **3단계 마이그레이션 전략**도 수립했다: 읽기 전환 → Feature Flag로 쓰기 전환 → 완전 전환. 각 단계에서 10분 내 롤백이 가능하도록 설계했다. 다만 이 전환은 아직 실행하지 않은 상태다. 설계까지만 하고, 실제 적용은 하지 못했다.

## 7. CSS도 모르는데 UI가 무너졌다

### 문제

기능을 빠르게 추가하다 보니 UI가 엉망이 됐다. 같은 버튼인데 페이지마다 모양이 다르고, 색상이 하드코딩(`Color(0xFF1A1A2E)`)되어 있고, 간격이 매직 넘버(`EdgeInsets.all(16)`)로 산재했다.

백엔드에서는 API 응답 형식을 `ApiResponse<T>`로 통일하면 일관성이 잡힌다. 그런데 UI 일관성은 그렇게 간단하지 않았다. 같은 "확인 버튼"인데 여기선 파란색이고 저기선 초록색이고, 어떤 데서는 테두리가 있고 어떤 데서는 없고. 처음에는 "그냥 예쁘면 되지 않나"라고 생각했는데, 화면이 10개를 넘어가니까 수습이 안 됐다.

### 해결: 두 가지 접근을 병행했다

**기존 코드에 디자인 토큰 적용**: 매직 넘버를 디자인 토큰(`AppSpacing.md`, `AppColors.surface`)으로 교체했다. 반복되는 UI 패턴은 컴포넌트로 추출했다 — `AppFormField`(입력 필드 공통화), `DialogHelpers`(다이얼로그 패턴 추출) 등으로 390줄을 절감했다.

**frontend_new에서 처음부터 다시**: 기존 코드에 토큰을 적용하면서도 한계를 느꼈다. 이미 만들어진 UI에 사후적으로 규칙을 입히려니 어디까지 바꿔야 할지 끝이 없었다. 그래서 `frontend_new/`에서 디자인 시스템을 먼저 구축(40개 시맨틱 컬러, 39개 컴포넌트)하고 그 위에 기능을 올리는 Bottom-up 접근을 실험했다. 컴포넌트 쇼케이스 페이지까지 만들었지만, 비즈니스 로직 연결까지는 가지 못했다.

결국 배운 건, **디자인 시스템은 나중에 덧붙이는 게 아니라 처음부터 깔아야 한다**는 것이다. 백엔드에서 `ApiResponse<T>`를 프로젝트 초반에 정하는 것처럼, UI의 색상·간격·컴포넌트 규칙도 첫 화면을 만들기 전에 정해야 한다.

## 8. 상태 관리 — 백엔드에는 없는 개념

### 문제

이건 프론트엔드에서 가장 힘들었던 부분이다.

Spring에서는 "상태 관리"라는 개념 자체가 거의 없다. 요청이 오면 DB에서 꺼내서 응답하고, 요청이 끝나면 아무것도 안 남는다. Stateless가 기본이다.

그런데 프론트엔드는 다르다. 인증 상태, 현재 선택한 그룹, 선택한 채널, 어디까지 읽었는지 — 이런 전역 상태들이 메모리에 계속 살아 있고, 서로 의존한다. Provider A(인증)가 바뀌면 B(사용자 정보)도 바뀌고, B가 바뀌면 C(그룹 목록)도 바뀌고, C가 바뀌면 D(채널)도 바뀌는 연쇄 갱신이 일어났다.

문제가 터진 건 로그아웃이었다. 로그아웃하면 모든 상태를 초기화해야 하는데, 상태들이 서로를 참조하고 있어서 순환 참조 버그가 발생했다. A를 초기화하려면 B가 필요하고, B를 초기화하려면 A가 필요한 상황이 됐다.

### 해결

Provider 간 **의존성을 단방향**으로 강제했다.

```
auth → user → groups → selectedGroup → workspace → channels → selectedChannel → posts
```

이 방향을 거스르는 참조는 금지했다. `selectedChannel`이 `auth`를 직접 참조하면 안 된다. 필요하면 상위에서 내려받아야 한다. 인증 로직은 별도 `AuthRepository`로 분리해서 순환을 끊었다.

Spring에서 `@Service` 간 순환 참조를 피하는 것과 비슷한 원리다. Spring에서도 A Service가 B Service를 참조하고 B가 다시 A를 참조하면 빈 생성이 안 되는데, 프론트엔드의 Provider에서도 같은 문제가 발생한다. 해결책도 비슷하다 — 의존 방향을 한쪽으로만 흐르게 하고, 공통 로직은 별도 모듈로 빼는 것.

`AsyncNotifier` 패턴으로 비동기 상태(로딩/에러/데이터)를 통합 관리했고, 선택 그룹이 바뀌면 하위 Provider가 자동으로 갱신되는 구조를 만들었다. Spring의 이벤트 리스너(`@EventListener`)와 비슷한 발상인데, 프론트엔드에서는 이런 반응형 갱신이 훨씬 자주 쓰인다.
