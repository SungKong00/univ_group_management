## 1. 그룹 계층 구조

### 해결하려는 문제

대학에는 공식 조직(학과, 연구실)과 비공식 조직(동아리, 스터디, 프로젝트팀)이 공존한다. 공식 조직은 대학 행정 체계를 따르고, 비공식 조직은 학생이 자유롭게 만들 수 있어야 한다. 이 두 가지를 하나의 구조로 표현하는 게 첫 번째 설계 과제였다.

### 설계 결정

**6가지 그룹 유형**으로 대학의 모든 조직을 표현한다:

| 유형 | 설명 | 생성 방식 |
|------|------|----------|
| UNIVERSITY | 대학교 (최상위) | 시스템 등록 |
| COLLEGE | 단과대 | 시스템 등록 |
| DEPARTMENT | 학과/학부 | 시스템 등록 |
| LAB | 연구실 | 교수 생성 |
| OFFICIAL | 공식 동아리/학회 | 학과 승인 필요 |
| AUTONOMOUS | 자율 그룹 (스터디, 프로젝트팀) | 학생 자유 생성 |

**핵심 원칙 — 그룹 독립성**: 상위 그룹이 하위 그룹의 내부를 감시하거나 통제할 수 없다. 유일한 권한은 "하위 그룹 생성 승인/거절"뿐이다. 대학이 동아리 내부 게시글을 검열하거나 멤버를 강제 추방할 수 없다는 뜻이다.

이 원칙은 처음부터 의도한 건 아니고, 구현하다 보니 "상위 그룹에 하위 그룹 접근 권한을 주면 권한 모델이 너무 복잡해진다"는 현실적인 이유도 있었다.

### 유저 시나리오

**시나리오: 학생이 스터디 그룹을 만드는 경우**
```
1. 학생이 "그룹 만들기" → 유형: AUTONOMOUS 선택
2. 그룹명, 설명, 카테고리 입력
3. 자율 그룹이므로 즉시 생성 (승인 불필요)
4. 워크스페이스 자동 생성 + 기본 채널 2개 생성
5. 생성자가 자동으로 그룹장 역할 부여
6. 다른 학생들이 "그룹 탐색"에서 검색 → 가입 신청
```

**시나리오: 공식 동아리 등록**
```
1. 학생이 "그룹 만들기" → 유형: OFFICIAL 선택
2. 소속 학과 선택 (AI/SW 학부)
3. 그룹명, 설명, 활동 계획 입력
4. 학과 관리자에게 승인 요청 전송
5. 학과 관리자가 승인 → 워크스페이스 자동 생성
6. 거절 시 → 사유와 함께 알림
```

### 구현 상태

- ✅ 6가지 그룹 유형 CRUD
- ✅ 계층 구조 (부모-자식 관계)
- ✅ 그룹 탐색 및 가입 신청
- ✅ 멤버 관리 (초대, 탈퇴, 추방)
- 🚧 그룹 탐색 고도화 (필터링, 추천)

## 2. 워크스페이스 & 채널

### 해결하려는 문제

하나의 그룹 안에서도 "공지는 읽기만", "자유게시판은 모두 작성 가능", "웹팀 채널은 웹팀만 접근" 같은 상황이 필요하다. 단순히 그룹 안에 게시판 하나만 있으면 이런 분리가 불가능하다.

### 설계 결정

**워크스페이스 = 그룹의 커뮤니케이션 허브**
- 그룹 승인 시 자동 생성 (1:1 관계)
- 그룹 멤버만 접근 가능

**채널 = 목적별 대화 공간**
- 기본 2개 채널이 템플릿으로 자동 생성:
  - 📢 공지사항 — 그룹장만 글 작성 가능
  - 💬 자유게시판 — 전체 멤버 글 작성 가능
- 그룹장이 추가 채널을 자유롭게 생성

**Secure by Default**: 새로 생성된 채널은 아무 권한도 없다. 그룹장이 명시적으로 "멤버 역할은 이 채널에서 읽기/쓰기 가능"이라고 설정해야 한다. 실수로 민감한 정보가 공개되는 것을 원천 차단한다.

### 채널 권한 예시

```
프로그래밍 동아리

채널: 📢 공지사항
  - 그룹장: 조회 ✅ 읽기 ✅ 쓰기 ✅ 댓글 ✅
  - 멤버:   조회 ✅ 읽기 ✅ 쓰기 ❌ 댓글 ✅

채널: 💻 웹개발팀
  - 그룹장:     조회 ✅ 읽기 ✅ 쓰기 ✅ 댓글 ✅
  - 웹팀 역할:  조회 ✅ 읽기 ✅ 쓰기 ✅ 댓글 ✅
  - 멤버:       조회 ❌ (채널 자체가 안 보임)
```

### 구현 상태

- ✅ 워크스페이스 자동 생성/삭제
- ✅ 채널 CRUD + 기본 템플릿
- ✅ 채널별 권한 바인딩 (ChannelRoleBinding)
- ✅ Secure by Default (새 채널 = 권한 0)

## 3. 2-Layer 권한 시스템

### 해결하려는 문제

단순 RBAC로는 "멤버 역할인데 공지 채널에서는 읽기만, 자유게시판에서는 쓰기도 가능" 같은 채널 단위 권한 차별화가 불가능하다. 그렇다고 채널마다 역할을 새로 만들면 관리가 폭발적으로 복잡해진다.

Spring Security의 `@PreAuthorize("hasRole('ADMIN')")` 같은 단순 체크로는 부족했다.

### 설계 결정: 하이브리드 2계층 모델

```
┌─────────────────────────────────┐
│  1계층: 시스템 역할 (그룹 단위)      │
│  ┌───────┐ ┌───────┐ ┌───────┐  │
│  │그룹장  │ │교수   │ │멤버   │  │
│  └───────┘ └───────┘ └───────┘  │
│          + 커스텀 역할 (운영진 등)  │
└──────────────┬──────────────────┘
               │ ChannelRoleBinding
┌──────────────▼──────────────────┐
│  2계층: 채널 권한 (채널 단위)       │
│  역할 × 채널 × 권한 = 바인딩       │
│                                  │
│  CHANNEL_VIEW  — 채널 보이기      │
│  POST_READ     — 게시글 읽기      │
│  POST_WRITE    — 게시글 쓰기      │
│  COMMENT_WRITE — 댓글 쓰기       │
│  FILE_UPLOAD   — 파일 업로드      │
└──────────────────────────────────┘
```

**그룹장은 항상 전체 권한** — 시스템 역할이 LEADER인 사용자는 모든 채널에서 모든 권한을 갖는다. 바인딩 없이도 자동 허용된다.

**커스텀 역할** — 그룹장이 "운영진", "모집담당" 같은 역할을 만들고, 특정 채널에 특정 권한을 바인딩할 수 있다.

### 성능 최적화: N+1 문제 해결

권한 체크는 모든 API 요청마다 일어나므로 성능이 중요하다.

**문제**: 사용자가 채널 목록을 볼 때, 각 채널마다 "이 사용자가 CHANNEL_VIEW 권한이 있는지" 확인해야 한다. 채널 10개면 10번 DB 조회 → N+1 문제.

**해결**:
1. **Fetch Join**: 사용자의 모든 채널 바인딩을 한 번에 로딩
2. **Caffeine 캐시**: 로딩된 권한 정보를 캐시에 저장 (TTL 5분)
3. **2단계 쿼리**: 시스템 역할 조회(1회) + 채널 바인딩 조회(1회) = 최대 2회

→ [N+1 해결 상세](technical-challenges.md#1-권한-확인할-때마다-db를-12번-찌르고-있었다)

### 감사 로깅

모든 권한 변경은 AuditLogger를 통해 기록된다. "누가 언제 어떤 채널에서 어떤 역할의 권한을 변경했는지" 추적할 수 있다.

### 구현 상태

- ✅ 시스템 역할 3종 (Leader, Professor, Member)
- ✅ 커스텀 역할 CRUD
- ✅ ChannelRoleBinding CRUD
- ✅ PermissionEvaluator (권한 판단 엔진)
- ✅ Caffeine 캐시 (PermissionCacheManager)
- ✅ AuditLogger (감사 로깅)
- ✅ 단위 테스트 21개 (PermissionEvaluatorTest)

## 4. 게시글 & 댓글

### 해결하려는 문제

게시글/댓글은 가장 기본적인 기능이지만, "읽은 위치 기억", "낙관적 업데이트", "권한 기반 접근 제어"가 결합되면 상당히 복잡해진다.

### 주요 기능

**게시글 고정(Pin)**: 채널에서 중요한 공지를 최상단에 고정한다. POST_MANAGE 권한이 있는 사용자만 고정/해제 가능하다.

**읽은 위치 추적**: 사용자가 채널을 떠났다 다시 들어오면, 마지막으로 읽은 게시글 이후부터 "안 읽은 글" 구분선이 표시된다. 처음에 시간 기반으로 만들었다가 부정확해서 Post ID 기반으로 재설계했다.

**Optimistic UI**: 글/댓글 작성 시 서버 응답을 기다리지 않고 즉시 화면에 반영한다. 서버에서 실패 응답이 오면 롤백하고 사용자에게 알림을 보여준다. 백엔드 개발자 입장에서는 "서버 확인도 안 하고 보여준다고?" 싶었는데, 사용자 경험을 위해 필요한 패턴이었다.

### Clean Architecture 적용

이 기능을 Clean Architecture로 재구성한 과정:

```
Domain Layer
├── entities/       PostEntity, CommentEntity
├── repositories/   IPostRepository, ICommentRepository (인터페이스)
└── value_objects/  PostContent, CommentContent

Data Layer
├── repositories/   PostRepositoryImpl, CommentRepositoryImpl
├── datasources/    PostRemoteDataSource, CommentRemoteDataSource
└── converters/     PostConverter, CommentConverter (Legacy 호환)

Presentation Layer
├── providers/      PostListNotifier, CommentListNotifier (AsyncNotifier)
├── widgets/        PostCard, CommentTile, PostEditor
└── pages/          PostListPage, PostDetailPage
```

백엔드에서 Controller → Service → Repository 3계층은 익숙한데, 프론트엔드의 Clean Architecture는 용어가 비슷하면서도 달랐다. 기존 코드에서 Provider가 API 호출과 UI 로직을 모두 담당하고 있었는데, 이를 3계층으로 분리하면서 각 계층의 단위 테스트가 가능해졌다. 29개 테스트를 작성해서 모든 계층을 검증했다.

### 구현 상태

- ✅ 게시글 CRUD + 고정(Pin)
- ✅ 2단계 댓글 (댓글 + 대댓글)
- ✅ 읽은 위치 추적 + 안 읽은 글 구분선
- ✅ Optimistic UI (작성/삭제)
- ✅ Clean Architecture 3계층 분리
- ✅ Legacy 호환 (CommentConverter)
- ✅ 단위 테스트 29개

## 5. 캘린더 3종 (개인 / 그룹 / 장소예약)

### 해결하려는 문제

대학 생활에서 일정 관리가 복잡한 이유: 개인 시간표, 동아리 행사, 강의실 예약이 각각 다른 도구에서 관리된다. 이걸 하나로 통합하면 "내 시간표에 동아리 행사가 겹치는지", "세미나실이 그 시간에 비어있는지"를 한눈에 볼 수 있다.

### 개인 캘린더

학생의 주간 시간표와 개인 일정을 관리한다.

- 주간/월간 뷰 전환
- 반복 일정 설정 (매주 월/수/금 등)
- 다른 캘린더(그룹)와 겹침 확인

### 그룹 캘린더

동아리/학회의 공식 행사와 비공식 모임을 관리한다.

**이벤트 유형**:

| 유형 | 생성자 | 특징 |
|------|--------|------|
| GENERAL (공식 전체) | 그룹장 | 모든 멤버 대상 |
| TARGETED (공식 지정) | 그룹장 | 특정 멤버만 대상 |
| RSVP (참석 확인) | 그룹장 | 참석 여부 회신 요청 |
| UNOFFICIAL (비공식) | 아무 멤버 | 개인 일정을 그룹에 공유 |

**RSVP 워크플로우**:
```
그룹장: "12/15 워크숍" 이벤트 생성 (RSVP 유형)
    ↓
멤버에게 알림 → 참석/불참/미정 선택
    ↓
그룹장: 참석 현황 확인 → 장소 예약 결정
```

### 장소 예약 시스템

강의실, 동아리방 같은 공용 장소의 예약을 관리한다.

**설계 포인트**:

- **운영시간**: 요일별로 다른 시간 설정 가능 (월~금 09:00-21:00, 토 10:00-17:00)
- **차단 시간**: 공휴일, 시험기간 등 예약 불가 시간 설정
- **이용 그룹**: 장소를 사용할 수 있는 그룹을 지정 (학과 소속 동아리만 등)
- **관리자**: 장소별 관리자 지정 + 관리자 이전 기능
- **예약 충돌**: 같은 시간대 중복 예약 자동 방지

**예약 워크플로우**:
```
멤버: 장소 목록에서 "세미나실 A" 선택
    ↓
캘린더 뷰에서 빈 시간 확인
    ↓
예약 요청 (날짜, 시간, 목적)
    ↓
시스템: 운영시간/차단시간/중복 체크
    ↓
예약 확정 → 캘린더에 표시
```

### 구현 상태

- ✅ 개인 캘린더 (프론트엔드 + 백엔드)
- ✅ 그룹 캘린더 — 백엔드 API 전체 (Phase 1-7)
- ✅ 장소 예약 — 백엔드 API 전체 (Phase 1)
- 🚧 그룹 캘린더 프론트엔드 (Phase 8-10 진행 중)
- ⏳ 장소 예약 프론트엔드 (Phase 2-5 대기)
- ⏳ 캘린더 간 자동 동기화 (향후)
- ⏳ 최적 시간 추천 (향후)

## 6. 모집 시스템

### 해결하려는 문제

동아리/학회가 새 멤버를 받을 때, 보통 SNS에 모집 공고를 올리고 구글 폼으로 지원을 받는다. 이러면 모집 공고 관리, 지원서 확인, 합격 통보, 그룹 멤버 추가가 전부 수동이다. 이 전체 과정을 시스템에서 한 번에 처리하도록 설계했다.

### 전체 워크플로우

```
1. 그룹장이 모집 공고 작성
   - 제목, 설명, 지원 양식 (자유 작성 or Q&A)
   - 선택: 마감일, 정원

2. 학생이 모집 공고 탐색
   - 그룹 탐색 화면에서 "모집 중" 필터
   - 모집 상세 확인 → 지원서 작성 → 제출

3. 그룹장이 지원서 검토
   - 지원자 목록 확인
   - 각 지원서에 승인/거절 + 피드백

4. 승인된 학생
   - 자동으로 그룹 멤버에 추가
   - 기본 "멤버" 역할 부여
   - 워크스페이스에 즉시 접근 가능

5. 모집 종료
   - 마감일 도래 시 자동 종료
   - 또는 그룹장이 수동 종료
   - 종료된 모집은 아카이브로 이동
```

### 설계 규칙

- **그룹당 활성 모집 1개**: 여러 모집을 동시에 열면 혼란 → 하나만 허용
- **1인 1지원**: 같은 모집에 중복 지원 불가. 철회 후 재지원은 가능
- **Soft Delete**: 지원서가 있는 모집 공고는 물리 삭제 대신 비활성화
- **자동 멤버 추가**: 승인 시 별도 초대 없이 즉시 멤버로 등록

### 구현 상태

- ✅ 모집 공고 CRUD (백엔드)
- ✅ 지원서 제출/철회 (백엔드)
- ✅ 승인/거절 + 자동 멤버 추가 (백엔드)
- ✅ 마감 자동 종료
- 🚧 모집 프론트엔드 UI (진행 중)

## 부가 기능

### Google OAuth2 인증

원클릭 로그인으로 진입 장벽을 낮추고, 서버에서 자체 JWT를 발급한다.

```
사용자 → Google 로그인 → ID Token 획득
    ↓
서버: Google ID Token 검증 → 사용자 조회/생성
    ↓
서버: Access Token (1시간) + Refresh Token (7일) 발급
    ↓
클라이언트: Access Token으로 API 호출
    ↓ (만료 시)
클라이언트: Refresh Token으로 자동 갱신
```

### 멤버 선택 시스템

그룹 캘린더에서 "특정 멤버만 대상으로 이벤트 생성" 같은 기능에 필요한 멤버 선택 UI를 3단계로 설계했다.

1. **필터 설정**: 역할, 그룹, 학년, 학번으로 필터
2. **미리보기**: 필터에 해당하는 멤버 목록 확인
3. **저장 방식 선택**:
   - DYNAMIC — 필터 조건 저장 (나중에 조건에 맞는 새 멤버도 자동 포함)
   - STATIC — 현재 목록을 고정 저장 (추가 멤버는 수동 포함)
