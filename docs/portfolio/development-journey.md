# 프로젝트 변천사

4개월간 이 프로젝트가 어떻게 진화해왔는지를 정리합니다. 결과물 자체보다는 "왜 당시 그러한 결정을 내렸는지"에 초점을 맞추었습니다.

---

## 9월: 실행 가능한 최소 기능 구현 (MVP)

첫 커밋은 9월 7일, Spring Boot 프로젝트 초기 설정으로 시작되었습니다.

초기 2주간의 목표는 속도였습니다. Google OAuth 로그인, 유저·그룹·워크스페이스 엔티티를 구축하고, 프론트엔드에서 로그인 → 그룹 선택 → 채널 진입으로 이어지는 기본 흐름을 완성했습니다. 백엔드는 17개의 컨트롤러가 하나의 패키지에 포함된 모놀리식 구조였으나, 우선적으로 "동작하는 결과물"을 만드는 데 집중했습니다.

Flutter는 이 시기에 처음으로 접하게 되었습니다. 백엔드(Spring) 개발에 익숙했던 저에게 프론트엔드 개발은 모든 것이 생소했습니다. 위젯이라는 개념, UI가 직접 상태를 보유한다는 점, 그리고 `build()` 메서드가 빈번하게 호출되는 구조 등은 Spring의 Controller가 요청을 처리하고 응답을 반환하는 방식과는 완전히 다른 세계였습니다.

이 시기에 내린 중요한 결정 중 하나는 **권한 시스템을 하드코딩하지 않고 엔티티 기반으로 설계한 점**입니다. 초기에는 `if (role == "ADMIN")`과 같은 임시 코드가 있었으나, 2주차에 `GroupPermissionEvaluator`와 `ChannelRoleBinding` 엔티티를 도입하며 구조를 확립했습니다. Spring Security를 학습하며 "권한은 유연하게 관리되어야 한다"는 원칙을 인지하고 있었기 때문이며, 결과적으로 이 결정은 프로젝트 후반부에 큰 도움이 되었습니다.

**9월 말 기준 구현 완료 기능**: 로그인, 그룹 CRUD, 워크스페이스/채널 탐색, 기본적인 권한 검증

## 10월: 기능 확장 및 복잡성 대응

10월은 가장 활발한 개발이 이루어진 달이었습니다 (총 331 커밋). 캘린더 시스템, 장소 예약, 그룹 모집 공고, 멤버 관리 기능을 집중적으로 구현했습니다.

### 권한 모델의 전면 개편 (10월 초)

기능이 확장됨에 따라 초기 권한 모델의 한계가 명확해졌습니다. "멤버" 역할이 모든 채널에서 동일한 권한을 갖는 것은 실제 운영 환경에 적합하지 않았습니다. 공지사항 채널은 읽기 전용으로, 자유게시판은 쓰기 권한까지 허용하는 등의 세밀한 제어가 필요했습니다.

이를 위해 **RBAC와 Channel Override를 결합한 하이브리드 모델**로 전환했습니다.

```
변경 전: 역할(그룹장/멤버) → 전역 권한 적용
변경 후: 역할(그룹장/교수/멤버) → 채널별 권한 바인딩 적용
```

핵심 원칙은 "새로운 채널은 명시적 권한 부여 전까지 접근이 차단된다"는 Secure by Default 방식이었습니다. 또한 그룹장, 교수, 멤버 등 시스템 역할 3종은 수정이 불가능한 불변 상태로 정의했습니다. 이는 Spring Security에서 기본 정책을 "모든 요청 거부"로 설정하는 것과 유사한 접근 방식이었습니다.

### 프론트엔드 코드 복잡도 관리의 필요성

기능을 빠르게 추가함에 따라 프론트엔드 코드의 결합도가 높아졌습니다. 동일한 성격의 UI 요소임에도 페이지마다 스타일이 상이하거나, 색상값이 하드코딩되고, 단일 위젯 파일이 300줄을 넘어가는 현상이 발생했습니다. 백엔드에서는 서비스 로직 분리가 비교적 명확했으나, UI 코드의 적절한 분리 지점을 판단하는 데는 더 많은 고민이 필요했습니다.

이 시점에 시스템 전반의 리팩터링을 진행하기로 결정했습니다.

## 10월 말 ~ 11월: 시스템 리팩터링 및 안정화

### 공통 컴포넌트 추출 (10/25 ~ 10/27)

반복되는 UI 패턴을 식별하여 재사용 가능한 컴포넌트로 독립시켰습니다.

- `AppFormField` (223줄) — 입력 필드 로직 공통화
- `AppInfoBanner` (242줄) — 안내 배너 디자인 표준화
- `DialogHelpers` (107줄) — 다이얼로그 호출 패턴 정형화
- `AppDialogTitle` (74줄) — 다이얼로그 타이틀 UI 통일

이를 통해 총 390줄의 코드를 절감했습니다. 백엔드에서 공통 로직을 추출하는 것과 유사하나, UI 컴포넌트는 시각적인 일관성까지 고려해야 했기에 더욱 세밀한 작업이 요구되었습니다.

### Navigator 2.0 선언적 라우팅 전환 (11/5 ~ 11/10)

이 프로젝트에서 기술적으로 가장 도전적이었던 과정입니다. Flutter의 기존 Navigator(push/pop) 방식을 선언적 방식인 Navigator 2.0으로 전면 교체했습니다.

**교체 배경**: 그룹 전환, URL을 통한 특정 채널 직접 접근, 뒤로가기 동작 시 상태가 불일치하는 문제가 지속적으로 발생했습니다. Spring에서는 URL이 곧 리소스의 상태를 대변하지만, 프론트엔드에서는 URL과 화면 스택, 사용자 권한 상태가 유기적으로 동기화되어야 함을 깨달았습니다.

**진행 과정**: 총 7단계에 걸쳐 점진적으로 이관을 수행했습니다.

1. WorkspaceRouterDelegate 설계 및 구현
2. 핵심 네비게이션 로직 이관
3. 상태 기반 라우팅 체계 구축
4. Context-Aware Group Switching (권한 기반 폴백) 구현
5. LRU 캐시를 활용한 스크롤 위치 보존 로직 추가
6. 디바운싱, 로딩 처리, 에러 상태 등 엣지 케이스 대응
7. 접근성 및 성능 검증 테스트 수행

**결과**: 98개의 테스트를 모두 통과하며 안정적인 라우팅 시스템을 확보했습니다. 그룹 전환 시 이전 스크롤 위치를 복원하고, 중복 클릭을 방지하며, 에러 발생 시 자동으로 재시도하는 등 견고한 사용자 경험을 제공하게 되었습니다.

### 정적 분석 경고 제로화 달성 (11/13 ~ 11/14)

약 600여 건의 로그를 분석하여 68건의 코드 개선을 진행했습니다. 지원이 중단된 API를 교체하고 Null Safety를 강화했으며, 로그 레이어를 체계적으로 분류하여 분석 경고를 완전히 제거했습니다. Spring에서 어노테이션 경고를 정리하는 과정과 유사했으나, Flutter의 특성에 맞춰 레이어별 로그 수준을 직접 정의하며 구조를 다듬었습니다.

## 11월 중순 ~ 말: Clean Architecture 도입 및 확장

### 프론트엔드 도메인별 구조 개편

게시글, 댓글, 채널 기능을 순차적으로 Clean Architecture(Domain → Data → Presentation) 구조로 전환했습니다.

백엔드의 3계층 구조와 용어는 비슷했으나 프론트엔드에 적용하는 과정은 새로웠습니다. Domain Layer의 유스케이스는 Spring의 Service와 유사한 역할을 수행하지만, 프론트엔드에서는 UI 상태 변화까지 고려해야 했기에 설계의 깊이가 더 깊어졌습니다.

**게시글 기능**을 시작으로 엔티티와 유스케이스를 정의하고, 리포지토리 구현체와 DTO 변환기를 구축했으며, Presentation Layer에 AsyncNotifier 패턴을 적용했습니다.

**댓글 기능**에서는 낙관적 UI 업데이트(Optimistic Update)를 핵심적으로 도입했습니다. 사용자가 댓글을 작성하면 서버 응답 전 UI에 즉시 반영하고, 실패 시에만 롤백하는 구조를 구현했습니다. 데이터 정합성을 중시하는 백엔드 사고방식에서 벗어나 사용자 경험을 최우선으로 고려하는 설계 방식을 수용했습니다.

**읽음 위치 추적 시스템**은 초기 시간 기반 방식의 한계를 극복하기 위해 Post ID 기반으로 재설계했습니다. 이 과정에서 위젯 해제 후에도 비동기 콜백이 실행되는 레이스 컨디션을 발견하여 수정했으며, 이를 통해 Flutter와 Spring의 비동기 모델 차이를 깊이 이해할 수 있었습니다.

### 디자인 시스템 중앙화 (11/21 ~ 11/29)

코드 전반의 하드코딩된 수치들을 디자인 토큰(`AppSpacing` 등)으로 대체하고 색상, 타이포그래피 등을 중앙에서 관리하도록 변경했습니다. 이는 백엔드에서 응답 형식을 통일하여 일관성을 확보하는 것과 동일한 맥락의 설계였습니다.

## 11월 말 ~ 12월: 아키텍처 고도화 시도

### frontend_new: 디자인 시스템 기반의 Bottom-up 개발 실험

기존 구조에 Clean Architecture를 적용하며 느꼈던 디자인 시스템 통합의 한계를 극복하기 위해, `frontend_new/` 프로젝트에서 디자인 시스템 중심의 재구축 실험을 진행했습니다. 40개의 시맨틱 컬러와 8단계 간격 스케일을 먼저 정의하고 이를 바탕으로 39개의 공통 컴포넌트를 구현하며 문서화했습니다. 비록 비즈니스 로직까지 완벽히 연결하지는 못했으나, 기초부터 탄탄히 쌓아 올리는 방식이 결과물의 일관성에 기여한다는 확신을 얻었습니다.

### backend_new: Clean Architecture 마이그레이션

기존 모놀리식 백엔드를 DDD와 Clean Architecture 기반으로 전면 재구성했습니다. 백엔드 영역인 만큼 보다 체계적으로 접근할 수 있었습니다.

- **Phase 0**: 엔티티 설계서 및 API 목록, 도메인 의존성 분석 완료
- **Phase 1-2**: 6개 핵심 도메인별 엔티티 및 서비스 구현 완료
- **Phase 3**: 권한 엔진 고도화(캐싱 및 감사 로깅) 완료
- **Phase 4-5**: 52개 REST API 엔드포인트 및 보안 설정 완료
- **Phase 6**: 155개 테스트 케이스 작성 및 검증 (통과율 86%)
- **Phase 7**: 점진적 마이그레이션 전략 수립 완료

'읽기 우선 전환 → Feature Flag 기반 쓰기 전환 → 완전 전환'으로 이어지는 리스크 최소화 전략을 설계했으며, 각 단계별로 즉각적인 롤백이 가능하도록 구상했습니다.

## 회고 및 시사점

4개월간의 여정을 통해 얻은 가장 큰 교훈은 **"초기 설계의 중요성"**입니다.

권한 시스템을 초기에 엔티티 기반으로 설계한 덕분에 후반부의 복잡한 확장 요구에도 유연하게 대처할 수 있었습니다. 반면 UI 스타일은 사후에 규칙을 적용하는 과정에서 많은 노력이 필요했음을 확인했습니다.

또한 **테스트 코드가 강력한 안전망**이 된다는 점을 실감했습니다. Navigator 2.0 전환과 같은 대규모 리팩터링을 자신 있게 진행할 수 있었던 원동력은 98개의 견고한 테스트 코드였습니다.

마지막으로 **과정을 통한 학습의 가치**입니다. `backend_new`와 `frontend_new`가 최종 프로덕션에 완벽히 통합되지는 않았으나, Clean Architecture와 디자인 시스템을 직접 설계하고 구현하며 얻은 실전 감각은 큰 자산이 되었습니다. 특히 프론트엔드 초심자로서 Navigator 2.0이나 낙관적 업데이트와 같은 고급 패턴을 체득한 것은 매우 의미 있는 성과였습니다.
