# 프로젝트 변천사

4개월간 이 프로젝트가 어떻게 바뀌어왔는지를 정리한다. 결과물보다는 "왜 그때 그 결정을 했는지"에 초점을 맞췄다.

---

## 9월: 일단 돌아가게 만들기

첫 커밋은 9월 7일, Spring Boot 프로젝트 초기 설정이었다.

처음 2주는 속도가 목표였다. Google OAuth 로그인, 유저·그룹·워크스페이스 엔티티를 만들고, 프론트엔드에서 로그인 → 그룹 선택 → 채널 진입까지의 기본 흐름을 완성했다. 백엔드는 17개 컨트롤러가 한 패키지에 들어가는 모놀리식 구조였지만, 우선 "동작하는 것"을 만드는 데 집중했다.

Flutter는 이때 처음 만졌다. 백엔드(Spring)만 공부하다가 프론트엔드를 하니까 모든 게 낯설었다. 위젯이라는 개념, 상태를 UI가 직접 들고 있다는 것, `build()` 메서드가 매번 호출된다는 것. Spring에서 `@Controller`가 요청을 받아서 처리하고 응답만 돌려주는 것과는 완전히 다른 세계였다.

이때 내린 결정 중 하나가 **권한을 하드코딩하지 않고 엔티티로 설계한 것**이다. 처음엔 `if (role == "ADMIN")` 같은 코드가 있었는데, 2주차에 `GroupPermissionEvaluator`와 `ChannelRoleBinding` 엔티티를 만들면서 구조를 잡았다. Spring Security를 배우면서 "권한은 유연하게 관리해야 한다"는 걸 알고 있었기 때문이다. 나중에 이 결정이 큰 도움이 됐다.

**9월 말 시점에 동작하던 것**: 로그인, 그룹 CRUD, 워크스페이스/채널 탐색, 기본 권한 체크

---

## 10월: 기능을 쌓다가 복잡도에 부딪히다

10월은 가장 활발한 달이었다 (331 커밋). 캘린더 시스템, 장소 예약, 그룹 모집 공고, 멤버 관리 등을 한꺼번에 구현했다.

### 권한 모델을 갈아엎다 (10월 초)

기능이 늘면서 초기 권한 모델의 한계가 드러났다. "멤버"가 모든 채널에서 같은 권한을 갖는 건 현실과 맞지 않았다. 공지사항은 읽기만, 자유게시판은 쓰기까지 — 이런 차이를 표현할 수 없었다.

그래서 **RBAC + Channel Override 하이브리드**로 전환했다.

```
변경 전: 역할(그룹장/멤버) → 전역 권한
변경 후: 역할(그룹장/교수/멤버) → 채널별 권한 바인딩
```

핵심은 "새 채널은 아무 권한 없이 시작"하는 Secure by Default 원칙이었다. 시스템 역할 3종(그룹장/교수/멤버)은 수정 불가(불변)로 만들었다. 이건 Spring Security에서 기본 정책을 "모든 요청 거부"로 두는 것과 같은 발상이다.

### 프론트엔드 코드가 점점 더러워지다

기능을 빠르게 추가하면서 프론트엔드 코드가 엉켜가기 시작했다. 같은 버튼인데 페이지마다 스타일이 다르고, 색상 값이 하드코딩되어 있고, 300줄짜리 위젯 파일이 늘어났다. 백엔드에서는 `@Service` 하나가 길어지면 분리하면 끝인데, UI 코드는 어디서 끊어야 할지 감이 안 왔다.

이 시점에서 리팩터링을 결심했다.

## 10월 말~11월: 리팩터링

### 컴포넌트 추출 (10/25~27)

반복되는 UI 패턴을 잡아서 재사용 가능한 컴포넌트로 분리했다.

- `AppFormField` (223줄) — 입력 필드 공통화
- `AppInfoBanner` (242줄) — 안내 배너 공통화
- `DialogHelpers` (107줄) — 다이얼로그 패턴 추출
- `AppDialogTitle` (74줄) — 다이얼로그 타이틀 표준화

총 390줄을 절감했다. 백엔드에서 공통 로직을 `BaseService`로 빼는 것과 비슷한데, UI 컴포넌트는 "보이는 것"까지 통일해야 해서 신경 쓸 게 더 많았다.

### Navigator 2.0 전환 (11/5~10)

이 프로젝트에서 가장 도전적이었던 작업이다. Flutter의 명령형 Navigator(push/pop)를 선언적 Navigator 2.0으로 전면 교체했다.

**왜 바꿨나**: 그룹을 전환하거나, URL로 특정 채널에 직접 접근하거나, 뒤로가기를 누를 때 상태가 꼬이는 문제가 반복됐다. Spring에서는 URL이 곧 상태라서 이런 문제가 없는데, 프론트엔드에서는 URL과 화면 스택과 사용자 상태가 따로 놀 수 있다는 걸 이때 깨달았다.

**어떻게 바꿨나**: 7단계로 나눠서 점진적으로 전환했다.

1. WorkspaceRouterDelegate 구현
2. 기본 네비게이션 이관
3. 상태 기반 라우팅 완성
4. Context-Aware Group Switching (권한 기반 폴백)
5. 스크롤 위치 보존 (LRU 캐시)
6. 엣지 케이스 (디바운싱, 로딩, 에러 상태)
7. 접근성 + 성능 테스트

**결과**: 98개 테스트 전부 통과. 그룹 전환 시 스크롤 위치를 기억하고, 빠른 연속 클릭을 무시하고, 에러 시 자동 재시도하는 안정적인 네비게이션이 완성됐다.

### Flutter 정적 분석 0개 달성 (11/13~14)

602개 로그를 감사(audit)하고 68건의 코드 개선을 수행했다. deprecated API 제거, null safety 강화, 로그 레이어 분류(UI/Provider/Repository/Service) 등을 정리해서 분석 경고를 완전히 없앴다. Spring에서 `@Deprecated` 경고를 정리하는 것과 비슷한 작업인데, Flutter는 로그 레벨 개념이 좀 달라서 직접 레이어를 나눠야 했다.

## 11월 중순~말: Clean Architecture 적용

### 프론트엔드 기능별 Clean Architecture

Post, Comment, Channel 기능을 하나씩 Clean Architecture(Domain → Data → Presentation)로 전환했다.

백엔드에서 Controller → Service → Repository 3계층은 익숙한데, 프론트엔드의 Clean Architecture는 용어가 비슷하면서도 달랐다. Domain Layer에 "유스케이스"라는 게 있는데, Spring의 Service 메서드와 비슷한 역할이다. 다만 프론트엔드에서는 UI 상태 변환까지 포함해야 해서 더 복잡했다.

**Post 기능**이 첫 번째 대상이었다. Domain Layer에 엔티티와 유스케이스를 정의하고, Data Layer에 리포지토리 구현과 DTO 변환기를 만들고, Presentation Layer에 AsyncNotifier 패턴을 적용했다.

**Comment 기능**은 Post 패턴을 따르되, 낙관적 UI 업데이트(Optimistic Update)를 핵심으로 추가했다. 댓글을 작성하면 서버 응답을 기다리지 않고 즉시 화면에 반영하고, 실패 시 롤백한다. 백엔드에서는 상상도 못 할 패턴인데, 사용자 경험을 위해 필요하다는 걸 이해하게 됐다. 29개 테스트로 이 흐름을 검증했다.

**Channel ReadPosition**은 "어디까지 읽었는지" 추적 시스템이다. 처음엔 시간 기반으로 만들었다가 부정확해서 Post ID 기반으로 재설계했다. 구현 과정에서 VisibilityDetector가 dispose된 후에도 비동기 콜백이 실행되는 Race Condition을 발견하고 수정했다. Flutter의 비동기 모델이 Spring의 `@Async`와 꽤 다르다는 걸 이때 제대로 체감했다.

### 디자인 시스템 중앙화 (11/21~29)

매직 넘버(`EdgeInsets.all(16)`)를 디자인 토큰(`AppSpacing.md`)으로 교체하고, 색상·타이포그래피·그림자·둥글기를 중앙 관리하도록 전환했다. 백엔드에서 `ApiResponse<T>` 래퍼로 응답 형식을 통일한 것과 같은 발상이다.

## 11월 말~12월: 더 나은 구조를 향한 시도

### frontend_new: 디자인 시스템부터 시작하기

기존 프론트엔드에 Clean Architecture를 적용하면서도 근본적인 한계를 느꼈다. 디자인 시스템이 사후에 덧붙여진 거라 완전히 통합되지 않았다.

그래서 `frontend_new/`에서 **디자인 시스템을 기반으로 처음부터 다시 시작**하는 실험을 했다. Linear 앱에서 영감을 받아 토큰 기반 디자인 시스템(40개 시맨틱 컬러, 8단계 간격 스케일)을 먼저 구축하고, 그 위에 39개 컴포넌트를 만들고, 컴포넌트 쇼케이스 페이지로 문서화했다.

비즈니스 로직 연결까지는 가지 못했지만, "디자인 시스템 → 컴포넌트 → 기능"이라는 Bottom-up 접근이 훨씬 일관된 결과물을 만든다는 확신을 얻었다.

### backend_new: Clean Architecture 마이그레이션

기존 모놀리식 백엔드를 `backend_new/`에서 DDD + Clean Architecture로 전면 재구성했다. 이건 내 전문 분야(백엔드)니까 상대적으로 수월했다.

- **Phase 0**: 29개 엔티티 설계서, 47개 API 목록, 도메인 의존성 그래프 작성
- **Phase 1-2**: 6개 도메인(User, Group, Permission, Workspace, Content, Calendar)별 엔티티 + 서비스 구현
- **Phase 3**: PermissionEvaluator에 캐싱과 감사 로깅 추가
- **Phase 4-5**: REST API 52개 엔드포인트 구현, JWT + OAuth2 보안
- **Phase 6**: 155개 테스트 작성 (133개 통과, 86%)
- **Phase 7**: 3단계 점진적 마이그레이션 전략 수립

마이그레이션 전략은 "읽기 전환 → Feature Flag로 쓰기 전환 → 완전 전환"의 3단계로 설계했다. 각 단계에서 10분 내 롤백이 가능하도록 했다. 실제 전환은 아직 실행하지 않았다.

## 돌아보며

4개월간 가장 많이 배운 건 **"처음에 잘 설계해야 한다"**는 것이었다.

권한 시스템은 초기에 엔티티로 설계해둔 덕분에 나중에 확장이 수월했다. Spring Security를 배우면서 "권한은 유연하게"라는 원칙을 알고 있었던 게 큰 도움이 됐다. 반면 UI는 빠르게 쌓다가 나중에 디자인 시스템을 적용하려니 어려웠고, 결국 새로 시작하는 쪽이 깔끔했다.

두 번째는 **테스트가 리팩터링의 안전망**이라는 것. Navigator 2.0 전환 같은 대규모 변경도 98개 테스트가 있었기에 자신 있게 진행할 수 있었다. Spring에서 JUnit으로 테스트를 짜는 건 배웠지만, 프론트엔드(위젯 테스트, 통합 테스트)에서도 이렇게 중요하다는 걸 체감했다.

세 번째는 **완성보다 과정이 중요하다**는 것. `backend_new`와 `frontend_new`는 통합되지 않았지만, 이 과정에서 Clean Architecture와 디자인 시스템에 대한 실전 감각을 쌓았다. 특히 프론트엔드를 처음 해본 입장에서, 4개월 만에 Navigator 2.0이나 Optimistic Update 같은 패턴을 구현할 수 있게 된 건 의미가 있었다.
