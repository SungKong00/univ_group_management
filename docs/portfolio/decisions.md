# 기술 선택과 회고

이 프로젝트는 저에게 첫 풀스택 도전이었습니다. 백엔드(Spring)만 공부하다가 프론트엔드를 처음 다루게 되었기에 모르는 부분이 많았습니다. 본 문서에는 왜 특정 기술을 선택했는지, 어떤 시행착오를 겪었는지, 그리고 다시 시작한다면 무엇을 개선하고 싶은지를 정리했습니다.

---

## 1. Kotlin + Spring Boot 선택 이유

### 선택

Spring Boot 3.5.5 + Kotlin 1.9.25 (코프링)

### 대안 비교

| 선택지 | 장점 | 단점 |
|--------|------|------|
| **Java + Spring** | 레퍼런스가 풍부하고 커뮤니티가 최대 규모임 | 보일러플레이트 코드가 많고 표현이 장황함 |
| **Kotlin + Spring** | 간결한 코드, Null Safety, Data Class 지원 | 한국어 자료가 상대적으로 부족함 |
| **Node.js (Express/NestJS)** | 프론트와 백엔드 언어 통일, 빠른 프로토타이핑 | 대규모 도메인 모델 구축에 부적합할 수 있음 |
| **Python (Django/FastAPI)** | 개발 속도가 빠르고 ORM이 강력함 | 타입 시스템이 상대적으로 약하고 구조화가 어려움 |

### 선택 근거

백엔드 프레임워크로서 Spring은 익숙한 선택이었습니다. 다만 Java가 아닌 Kotlin을 선택한 구체적인 이유가 있습니다. 이 프로젝트 이전에 동일한 주제로 Java + Spring Boot를 시도한 적이 있었는데, 당시 AI 에이전트와 협업하며 개발을 진행했습니다. 문제는 Java 코드가 너무 길어 엔티티 하나에도 Getter, Setter, Constructor, Builder 등이 과도하게 붙는 점이었습니다. AI 에이전트가 이를 모두 읽어야 하므로 컨텍스트 윈도우가 빠르게 소모되어 코드 이해 능력이 저하되는 현상이 발생했습니다.

Kotlin으로 전환하니 Data Class 한 줄이 Java POJO 50줄을 대체할 수 있었습니다. 이를 통해 AI 에이전트와의 협업 효율이 비약적으로 향상되었으며, Null Safety 기능은 안정성을 더해주었습니다.

### 회고

잘한 점:
- Data Class를 활용하여 29개의 엔티티를 간결하게 정의할 수 있었습니다.
- Null Safety 덕분에 런타임 NPE(NullPointerException)가 단 한 번도 발생하지 않았습니다.
- MockK를 사용하여 테스트 코드를 Kotlin 스타일로 깔끔하게 작성할 수 있었습니다.

아쉬운 점:
- JPA와 Kotlin Data Class의 궁합이 기대만큼 완벽하지는 않았습니다 (15번 항목에서 상세 기술).
- 한국어 자료가 Java Spring에 비해 부족하여 문제 발생 시 영어 문서를 주로 참조해야 했습니다.

## 2. Flutter Web 선택 이유

### 선택

Flutter 3.9 (Web 타겟)

### 대안 비교

| 선택지 | 장점 | 단점 |
|--------|------|------|
| **React + TypeScript** | 웹 생태계 최강, 방대한 학습 자료 | 모바일 앱 별도 개발 필요 |
| **Next.js** | SSR/SSG 지원, SEO 최적화에 유리 | 본 프로젝트에서는 SEO의 중요도가 낮음 |
| **Flutter Web** | 크로스플랫폼 지원, 강력한 위젯 시스템 | 웹에서의 성능이 아직 일부 제한적임 |
| **React Native** | 모바일과 웹 동시 지원, JS 생태계 활용 | 웹 지원 수준이 Flutter에 비해 다소 약함 |

### 선택 근거

프론트엔드 기술을 깊이 있게 비교하기에는 경험이 부족한 상태였습니다. React나 Vue를 사용해본 적이 없었기에 익숙한 백엔드 지식을 활용할 수 있는 방안을 모색했습니다.

Flutter를 선택한 이유는 다음과 같습니다:

1. **Dart와 Kotlin의 유사성**: 타입 시스템, Async/Await, 클래스 기반 OOP 등 언어적 특징이 비슷하여 문법 적응에 어려움이 없을 것이라 판단했습니다.
2. **모바일 확장 가능성**: 대학생 대상 서비스이므로 모바일 접근성이 중요한데, Flutter를 통해 동일한 코드로 웹과 모바일을 모두 대응할 수 있다는 점이 매력적이었습니다.
3. **직관적인 위젯 시스템**: HTML/CSS 지식이 부족한 상태에서 위젯을 조합하여 화면을 구성한다는 개념이 이해하기 쉬웠습니다.

### 프론트엔드 개발 입문기

Spring에서는 Controller에 어노테이션을 추가하고 Service 로직을 처리한 뒤 응답을 보내는 흐름이 명확했습니다. 상태값은 주로 DB에 보관되었습니다.

하지만 Flutter는 화면과 코드가 유기적으로 결합된 구조였습니다. "상태의 변화가 UI의 재빌드로 이어진다"는 개념을 이해하는 데 시간이 다소 소요되었습니다. 백엔드와 달리 프론트엔드에서는 화면이 "살아있는 상태"를 유지하며, 상태 변경 시마다 위젯 트리가 다시 그려지는 과정을 추적해야 했습니다. 처음에는 성능 저하를 우려했으나, Flutter가 차이(Diff)를 계산하여 변경된 부분만 효율적으로 갱신한다는 점을 배우며 구조를 파악할 수 있었습니다.

### 회고

잘한 점:
- Dart 언어가 Kotlin과 비슷하여 빠르게 적응할 수 있었습니다.
- 위젯 단위 설계를 통해 자연스럽게 컴포넌트를 분리할 수 있었습니다.
- `flutter_test`를 활용하여 300개 이상의 위젯 테스트를 작성하며 안정성을 확보했습니다.

아쉬운 점:
- Flutter Web의 초기 로딩 속도가 상대적으로 느립니다 (WASM 빌드 도입 등의 개선 여지가 있습니다).
- 드래그 앤 드롭이나 리치 텍스트 에디터 등 웹 특화 라이브러리가 React 등에 비해 적습니다.
- 프론트엔드 개발 자체가 처음이다 보니 직면한 문제가 기술적 한계인지, 개념적 미숙함인지 구분하기 어려운 경우가 있었습니다.

## 3. 데이터베이스 전략: H2 + PostgreSQL

### 선택

개발 환경: H2 (인메모리) / 운영 환경: PostgreSQL

### 선택 근거

6개 도메인에 걸친 29개의 엔티티가 복잡하게 얽혀 있는 구조이므로 관계형 데이터베이스가 적합하다고 판단했습니다. 특히 다대다 관계 처리가 중요한 권한 모델 설계에서 RDB는 익숙하고 강력한 도구였습니다.

개발 단계에서는 H2 인메모리 방식을 사용하여 서버 기동 시 스키마가 자동 생성되도록 설정함으로써 개발 속도를 높였습니다. 별도의 DB 설치 없이 즉시 개발을 시작할 수 있는 환경을 구축했습니다.

### 회고

잘한 점:
- `spring.jpa.open-in-view: false` 설정을 통해 LazyLoading 관련 이슈를 사전에 방지했습니다.
- 로컬 DB 설치 없이 즉시 실행 가능한 환경이 개발 효율성을 크게 높였습니다.

아쉬운 점:
- Flyway나 Liquibase와 같은 마이그레이션 도구를 초기에 도입하지 못한 점이 아쉽습니다. `ddl-auto: create-drop` 방식은 운영 환경에서 리스크가 크므로 개선이 필요합니다.
- H2와 PostgreSQL 간의 미세한 SQL 방언 차이가 향후 잠재적 위험 요소가 될 수 있음을 인지했습니다.

## 4. 인증 시스템: Google OAuth 2.0 + 자체 JWT

### 선택

Google OAuth 2.0 로그인 연동 후 자체 JWT (Access 24h / Refresh 7d) 발급

### 선택 근거

대학 환경 특성상 대다수 학생이 구글 계정을 보유하고 있어 Google OAuth는 진입 장벽을 낮추는 최선의 선택이었습니다.

자체 JWT를 발급하는 방식을 채택한 이유는 토큰 정보를 직접 제어하기 위해서입니다. 사용자의 그룹 멤버십이나 역할 정보를 클레임(Claim)에 포함하여, 매 요청 시 DB 조회 없이도 기본적인 권한 판단이 가능하도록 설계했습니다.

### 시행착오: 리프레시 토큰의 진화 과정

**1단계: Access Token 단일 운용**
초기에는 하나의 토큰만 사용했으나, 보안과 사용자 편의성 사이에서 만료 시간 설정의 딜레마를 겪었습니다.

**2단계: Access + Refresh (단순 갱신)**
리프레시 토큰을 도입했으나, 리프레시 토큰 자체의 만료 기간이 지나면 다시 로그인해야 하는 불편함이 있었습니다.

**3단계: Access + Refresh (슬라이딩 윈도우)**
최종적으로 리프레시 토큰 사용 시마다 새로운 리프레시 토큰을 발급하는 방식을 적용했습니다. 이를 통해 활발히 사용하는 사용자는 재로그인 없이 서비스를 계속 이용할 수 있도록 개선했습니다.

백엔드 구현은 Spring Security와 JWT 라이브러리를 통해 수월하게 진행했으나, 프론트엔드에서 토큰을 안전하게 저장하고 Dio 인터셉터를 활용해 자동으로 갱신하는 비동기 흐름을 구현하는 과정이 새로운 도전이었습니다.

## 5. 권한 시스템: RBAC + Channel Override

### 선택

2계층 권한 모델 (그룹 수준 RBAC + 채널 수준 권한 오버라이드)

### 발전 과정

**1단계: 하드코딩 방식 (초기)**
역할이 단순했던 초기에는 단순한 조건문으로 처리했습니다.

**2단계: 엔티티 기반 RBAC**
역할과 권한을 별도의 엔티티로 분리하여 DB에서 관리하기 시작했습니다. 이 결정이 향후 확장성의 밑거름이 되었습니다.

**3단계: RBAC + Channel Override (최종)**
채널별로 상이한 권한 요구사항을 반영하기 위해 `ChannelRoleBinding` 엔티티를 추가했습니다. "Secure by Default" 원칙에 따라 새 채널은 명시적 권한 부여 전까지 접근이 차단되도록 설계했습니다.

### 성능 최적화

권한 확인은 시스템 전반의 성능에 직결되므로 Fetch Join을 통해 N+1 문제를 해결하고, Caffeine 캐시를 적용하여 DB 조회 횟수를 최소화했습니다. JPA 성능 최적화 경험이 큰 도움이 된 영역이었습니다.

## 6. 상태 관리: Riverpod 도입 배경

### 선택

flutter_riverpod 2.6 + AsyncNotifier + Family 패턴

### 학습의 어려움

백엔드에서는 Stateless한 구조가 기본이기에 상태 관리라는 개념이 생소했습니다. 반면 Flutter에서는 현재 선택된 그룹, 로딩 상태, 에러 발생 여부 등을 코드에서 실시간으로 관리해야 했습니다.

초기에는 위젯 내부에 로직을 집중시켰으나 코드가 빠르게 복잡해졌습니다. BLoC는 구조는 견고하지만 보일러플레이트가 많아 부담스러웠습니다. Riverpod의 `AsyncNotifier`는 비동기 상태를 선언적으로 관리할 수 있어 Spring의 Service 레이어와 유사한 느낌으로 접근할 수 있었습니다.

### 시행착오: 순환 참조 해결

상태 간 연쇄적인 갱신 과정에서 로그아웃 시 순환 참조 버그가 발생했습니다. 이를 해결하기 위해 의존성 방향을 단방향으로 엄격히 제한했습니다:
`auth → user → groups → selectedGroup → workspace → channels → selectedChannel → posts`

이 원칙을 통해 구조를 명확히 하고 인증 로직을 분리하여 안정성을 확보했습니다. 백엔드의 레이어 분리 원칙이 프론트엔드에서도 유효함을 깨닫는 계기가 되었습니다.

## 7. 네비게이션: GoRouter 및 Navigator 2.0

### 선택

GoRouter 14.6 (Navigator 2.0 기반)

### 전환 이유

Spring의 단순한 URL 매핑과 달리 프론트엔드에서는 URL과 화면 스택, 권한 상태가 유기적으로 연결되어야 했습니다. 초기 Navigator 1.0 방식에서는 딥 링크나 그룹 전환 시 화면 스택이 꼬이는 문제가 잦았습니다.

Navigator 2.0으로 전환하며 선언적 라우팅을 구현했습니다. 특히 권한이 없는 그룹 접근 시 이전 그룹으로 안전하게 되돌리는 폴백 로직을 구현하는 데 집중했습니다. 98개의 테스트를 통해 다양한 시나리오를 검증하며 견고한 네비게이션 구조를 확립했습니다.

## 10. 테스트 전략: 60/30/10 피라미드

### 선택

통합 테스트 60% / 단위 테스트 30% / E2E 테스트 10%

### 비중 설정 이유

Spring Context 내에서 JPA 쿼리와 트랜잭션, 캐시 로직이 실제 DB와 유기적으로 동작하는지 검증하는 것이 가장 중요하다고 판단했습니다. 따라서 단순 로직 검증인 단위 테스트보다 통합 테스트의 비중을 높게 설정했습니다.

### 현재 현황

백엔드에서는 155개의 테스트를 작성하여 86% 이상의 통과율을 기록 중입니다. 일부 테스트 간 간섭 문제는 향후 테스트 격리 개선을 통해 해결할 계획입니다. 프론트엔드에서도 위젯 테스트를 포함한 300개 이상의 테스트를 통해 UI와 네비게이션 로직의 신뢰성을 확보했습니다.

---

## 11. 아키텍처 진화: Clean Architecture로의 여정

### 발전 단계

초기 모놀리식 구조에서 시작하여 도메인 간 결합도를 낮추기 위해 패키지를 분리했습니다. 최종적으로는 `backend_new/` 프로젝트를 통해 완벽한 Clean Architecture 구조를 설계했습니다. 도메인 간 통신은 반드시 인터페이스를 통하도록 강제하여 유연성을 확보했습니다. Modular Monolith 구조를 통해 현재 수준에서 불필요한 마이크로서비스의 복잡도는 피하면서 향후 확장 가능성을 열어두었습니다.

---

## 12. 코드 생성 도구: Freezed 및 json_serializable

### 도입 이유

Dart 언어의 장황한 보일러플레이트 코드를 줄이기 위해 Freezed를 도입했습니다. Kotlin의 Data Class와 유사한 편의성을 제공받을 수 있었습니다. JSON 직렬화 또한 자동 생성을 통해 실수를 줄이고 개발 속도를 향상시켰습니다.

---

## 13. 디자인 시스템 구축 실험

### 시사점

기능 구현 후 스타일을 입히는 방식(Top-down)과 디자인 시스템을 먼저 구축하는 방식(Bottom-up)을 모두 시도했습니다. 결과적으로 디자인 시스템은 프로젝트 초기 단계에서 확립해야 하는 기초 설계임을 깨달았습니다. UI 일관성을 확보하는 것이 백엔드 API 규격을 맞추는 것만큼 중요하다는 점을 학습했습니다.

---

## 14. 읽기 위치 추적 로직의 개선

### 재설계 과정

시간 기반에서 Post ID 기반으로 전환하여 정합성 문제를 해결했습니다. 또한 `VisibilityDetector`를 활용해 실제 게시글이 화면에 노출되는 시점을 포착하여 읽음 처리하는 반응형 UI를 구현했습니다. 이 과정에서 프론트엔드 특유의 생명주기 관리와 비동기 처리의 중요성을 체감했습니다.

---

## 15. JPA와 Kotlin 사용 시 주의사항

### 시행착오

Kotlin의 불변성(val) 원칙과 JPA의 변경 감지(Dirty Checking) 기능 사이의 트레이드오프를 경험했습니다. 결과적으로 엔티티의 업데이트가 필요한 필드는 `var`로 선언하여 JPA가 정상적으로 작동하도록 수정했습니다. Kotlin의 언어적 철학과 프레임워크의 동작 방식 사이의 균형을 맞추는 법을 배웠습니다.

---

## 16. 점진적 마이그레이션 전략

### 설계 내용

시스템 전체를 한 번에 교체하는 리스크를 방지하기 위해 '읽기 우선 전환 → Feature Flag 기반 쓰기 전환 → 완전 전환'이라는 3단계 전략을 수립했습니다. 비록 시간 관계상 완전 통합까지는 이르지 못했으나, 실제 대규모 시스템 개편에서 발생할 수 있는 문제들을 미리 고민하고 문서화한 것 자체가 값진 학습 경험이 되었습니다.

---

## 17. 낙관적 UI 적용과 사용자 경험

### 학습 내용

서버 응답 전 UI를 먼저 갱신하는 낙관적 업데이트를 통해 체감 성능을 향상시켰습니다. 백엔드 중심의 데이터 정합성 사고방식에서 벗어나, 사용자 경험을 최우선으로 고려하는 프론트엔드 관점을 수용하는 계기가 되었습니다. `AsyncNotifier`를 통해 상태 갱신 로직을 체계화하며 정교한 UI 인터랙션을 구현했습니다.

## 전체 회고

### 주요 성과

1. **권한 시스템의 확장성 확보**: 초기에 엔티티 기반으로 설계한 덕분에 복잡한 요구사항 변경에도 유연하게 대처할 수 있었습니다.
2. **Kotlin 도입의 효율성**: 코드량을 획기적으로 줄여 유지보수성과 개발 속도를 동시에 잡았습니다.
3. **견고한 테스트 코드**: 대규모 리팩터링 과정에서 테스트 코드가 강력한 안전망 역할을 수행했습니다.

### 아쉬운 점 및 개선 과제

1. **디자인 시스템의 조기 도입**: 처음부터 디자인 토큰과 컴포넌트 기반으로 접근했다면 UI 일관성 확보가 더 수월했을 것입니다.
2. **DB 마이그레이션 도구 미비**: Flyway 등을 통한 체계적인 DB 관리가 누락된 점이 아쉽습니다.
3. **신규 아키텍처 완전 통합**: `backend_new`의 성과를 실제 프로덕션에 완벽히 녹여내지 못한 점은 향후 과제로 남겨두었습니다.

### 다시 시작한다면

- 디자인 시스템(토큰 및 컴포넌트)을 최우선으로 구축하겠습니다.
- Flyway를 프로젝트 초기 단계에 설정하겠습니다.
- 테스트 격리를 강화하여 Flaky 테스트가 발생하지 않도록 개선하겠습니다.
- OpenAPI 스펙을 먼저 정의하여 프론트와 백엔드 간 협업 효율을 극대화하겠습니다.
- 프론트엔드 상태 관리 구조를 프로젝트 시작 전에 면밀히 설계하겠습니다.
