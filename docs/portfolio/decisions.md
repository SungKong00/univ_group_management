# 기술 선택과 회고

이 프로젝트는 나한테 첫 풀스택 도전이었다. 백엔드(Spring)만 공부하다가 프론트엔드를 처음 만졌고, 모르는 게 당연히 많았다. 여기에는 왜 이 기술을 골랐는지, 어디서 삽질했는지, 다시 하면 뭘 바꿀 건지를 정리했다.

---

## 1. 왜 Kotlin + Spring Boot인가

### 선택

Spring Boot 3.5.5 + Kotlin 1.9.25 (코프링)

### 대안

| 선택지 | 장점 | 단점 |
|--------|------|------|
| **Java + Spring** | 레퍼런스 풍부, 커뮤니티 최대 | 보일러플레이트 과다, 코드가 장황 |
| **Kotlin + Spring (코프링)** | 간결한 코드, null safety, data class | 한국어 자료 상대적으로 적음 |
| **Node.js (Express/NestJS)** | 프론트·백 같은 언어, 빠른 프로토타이핑 | 대규모 도메인 모델에 부적합 |
| **Python (Django/FastAPI)** | 빠른 개발, ORM 강력 | 타입 시스템 약함, 구조화 어려움 |

### 왜 골랐나

사실 이건 고민할 것도 없었다. Spring이 내가 아는 유일한 백엔드 프레임워크였으니까.

다만 Java가 아닌 Kotlin을 고른 이유가 있다. 이 프로젝트 전에 같은 주제로 Java + Spring Boot를 시도한 적이 있는데, 그때 AI 에이전트랑 같이 개발했다. 문제는 Java 코드가 너무 길다는 거였다. 엔티티 하나에 getter/setter/constructor/builder가 줄줄이 붙고, 간단한 DTO도 수십 줄이 됐다. 에이전트가 이걸 다 읽어야 하니까 컨텍스트 윈도우가 금방 차서, 코드 이해 능력이 눈에 띄게 떨어졌다.

Kotlin으로 바꾸니까 data class 한 줄이 Java POJO 50줄을 대체했다. AI 에이전트와의 협업 효율이 확 올랐고, null safety는 덤이었다.

### 돌아보면

잘한 점:
- data class로 29개 엔티티를 간결하게 정의할 수 있었다
- null safety 덕에 런타임 NPE가 한 번도 안 났다
- MockK로 테스트 코드도 Kotlin답게 깔끔하게 나왔다

아쉬운 점:
- JPA와 Kotlin data class 궁합이 생각만큼 좋지 않았다 (15번에서 자세히 다룸)
- 코프링 한국어 자료가 Java Spring에 비해 확실히 적어서, 문제 생기면 영어 문서를 뒤져야 했다

## 2. 왜 Flutter Web인가

### 선택

Flutter 3.9 (Web 타겟)

### 대안

| 선택지 | 장점 | 단점 |
|--------|------|------|
| **React + TypeScript** | 웹 생태계 최강, 자료 풍부 | 모바일 별도 개발 필요 |
| **Next.js** | SSR/SSG, SEO 좋음 | 이 프로젝트는 SEO 불필요 |
| **Flutter Web** | 크로스플랫폼, 위젯 시스템 | 웹 성능 아직 제한적 |
| **React Native** | 모바일+웹, JS 생태계 | 웹 지원이 Flutter보다 약함 |

### 왜 골랐나

솔직히 말하면, 프론트엔드 기술을 비교할 능력이 없었다. React도 Vue도 써본 적 없으니까. 내가 아는 건 Spring뿐이었다.

Flutter를 고른 이유는 세 가지였다:

1. **Dart가 Kotlin이랑 비슷했다.** 타입 시스템이 있고, async/await가 있고, 클래스 기반 OOP다. 적어도 언어 문법에서 헤매지는 않겠다고 생각했다.
2. **모바일 확장이 가능했다.** 대학생이 쓰는 서비스라 모바일이 중요한데, Flutter면 같은 코드로 웹이랑 모바일을 둘 다 커버할 수 있었다.
3. **"모든 게 위젯"이라는 설명이 직관적이었다.** HTML/CSS를 모르는 입장에서, 위젯을 조합해서 화면을 만든다는 개념이 이해가 됐다. (물론 실제로 해보니 그렇게 간단하지 않았지만.)

### 백엔드 개발자가 프론트엔드를 처음 만지면

Spring에서는 Controller에 `@GetMapping` 붙이면 끝이다. 요청이 오면 Service에서 처리하고 응답 보내면 된다. 상태? DB에 있다.

Flutter는 달랐다. 화면이 곧 코드고, 코드가 곧 화면이다. "버튼을 누르면 상태가 바뀌고, 상태가 바뀌면 화면이 다시 그려진다" — 이 개념이 처음에 잘 안 들어왔다. Spring에서는 서버가 응답을 보내면 서버 쪽 일은 끝이다. 근데 프론트엔드에서는 화면이 "살아있다". 상태가 바뀔 때마다 위젯 트리가 다시 빌드되고, 어디서 rebuild가 일어나는지 추적해야 한다.

`build()` 메서드가 매 상태 변경마다 호출된다는 걸 처음 알았을 때 "이러면 성능이 안 나오지 않나?" 싶었다. Flutter가 diff를 계산해서 바뀐 부분만 다시 그린다는 걸 이해하기까지 시간이 좀 걸렸다.

### 돌아보면

잘한 점:
- Dart가 Kotlin이랑 비슷해서 언어 적응이 빨랐다
- 위젯 단위 설계 덕에 컴포넌트 분리가 자연스러웠다
- flutter_test로 위젯 테스트를 300개 넘게 작성할 수 있었다

아쉬운 점:
- Flutter Web 초기 로딩이 느리다 (WASM 빌드가 아직 실험적이라 어쩔 수 없었다)
- 웹 특화 라이브러리가 확실히 적다 (드래그앤드롭, 리치 텍스트 에디터 등)
- 프론트엔드를 처음 하니까 "이게 Flutter 특유의 어려움인지, 프론트엔드 자체가 원래 이런 건지" 구분이 안 됐다

## 3. 데이터베이스: H2 + PostgreSQL 이중 전략

### 선택

개발환경 H2 (인메모리) + 운영환경 PostgreSQL

### 대안

| 선택지 | 장점 | 단점 |
|--------|------|------|
| **H2 + PostgreSQL** | 개발 빠름, 운영 안정 | 개발·운영 DB 차이 |
| **PostgreSQL only** | 환경 동일 | 로컬 설치 필요, Docker 의존 |
| **MySQL** | 대중적, 자료 풍부 | JSON 지원이 약함 |
| **MongoDB** | 스키마 유연, JSON 네이티브 | 복잡한 관계 쿼리에 부적합 |

### 왜 골랐나

6개 도메인에 29개 엔티티가 서로 복잡하게 엮여 있다. `ChannelRoleBinding`처럼 다대다 관계가 핵심인 모델에서는 관계형 DB가 자연스러웠다. 이건 Spring + JPA 하면서 익숙했던 부분이라 큰 고민 없이 결정했다.

개발할 때는 H2 인메모리로 서버 시작하면 스키마가 자동 생성되니까 편했다. DB 따로 설치할 필요 없이 바로 개발에 들어갈 수 있었다.

### 돌아보면

잘한 점:
- `spring.jpa.open-in-view: false`로 LazyLoading 이슈를 처음부터 차단했다
- 개발 시 DB 설치 없이 즉시 실행 가능한 게 개발 속도에 크게 도움됐다

아쉬운 점:
- **마이그레이션 도구(Flyway/Liquibase)를 안 넣었다.** `ddl-auto: create-drop`에 의존했는데, 운영 환경에서는 이러면 안 된다. 이건 확실히 초기에 넣었어야 했다.
- H2와 PostgreSQL SQL 방언 차이가 잠재적 위험이다 (아직 터지지는 않았지만)

## 4. 인증: Google OAuth 2.0 + 자체 JWT

### 선택

Google OAuth 2.0 로그인 → 자체 JWT (Access 24h + Refresh 7d) 발급

### 대안

| 선택지 | 장점 | 단점 |
|--------|------|------|
| **Google 토큰 직접 사용** | 구현 간단 | 토큰 수명·클레임 제어 불가 |
| **자체 JWT 발급** | 수명·클레임 완전 제어 | 직접 구현 필요 |
| **세션 기반** | 서버에서 상태 관리 | Stateless와 안 맞음 |
| **Firebase Auth** | 간편 | 벤더 종속, 세밀한 제어 어려움 |

### 왜 골랐나

Google OAuth는 대학 환경이니까 당연했다. 대부분 학생이 `@학교.ac.kr` 구글 계정을 갖고 있어서 별도 회원가입이 필요 없다.

자체 JWT를 발급하는 이유는 토큰에 담는 정보를 직접 제어하기 위해서다. 사용자의 그룹 멤버십이나 역할 정보를 클레임에 넣으면, 매 요청마다 DB를 조회하지 않아도 기본적인 인가 판단이 가능하다. 이 부분은 Spring Security 쪽이라 비교적 편하게 구현할 수 있었다.

### 시행착오: 리프레시 토큰 3번의 변화

**1차 — Access Token 하나만**

처음에는 Access Token 하나만 발급했다. 만료 시간을 24시간으로 길게 잡았는데, 탈취되면 24시간 동안 악용될 수 있다는 문제가 있었다. 짧게 잡으면 사용자가 자꾸 재로그인해야 하고.

**2차 — Access + Refresh (단순 갱신)**

Refresh Token을 추가했다. Access 만료 시 Refresh로 갱신하는 방식. 그런데 Refresh Token은 갱신 안 하고 그대로 두니까, 7일 후에 Refresh도 만료되면 다시 로그인해야 했다.

**3차 — Access + Refresh (슬라이딩 윈도우)**

최종적으로 Refresh Token을 쓸 때마다 새 Refresh Token도 같이 발급하는 슬라이딩 윈도우 방식을 적용했다. 7일 이내에 한 번이라도 앱을 쓰면 자동 갱신된다.

```
Access Token: 24시간 (짧게 — 탈취 시 피해 제한)
Refresh Token: 7일 (길게 — UX 유지)
Refresh 시: 새 Access + 새 Refresh 동시 발급
```

이 구현은 Spring Security + JWT 조합이라 백엔드 쪽은 수월했다. 오히려 어려웠던 건 **프론트엔드에서 토큰을 어디에 저장하고 어떻게 자동 갱신할지**였다. `flutter_secure_storage`로 저장하고 Dio 인터셉터에서 만료 체크 후 자동 갱신하는 흐름을 만들었는데, 이게 프론트엔드에서 비동기 처리하는 첫 경험이었다.

## 5. 권한 시스템: RBAC + Channel Override

### 선택

2계층 권한 모델 — 그룹 수준 RBAC + 채널별 ChannelRoleBinding

### 대안

| 선택지 | 장점 | 단점 |
|--------|------|------|
| **단순 RBAC** | 이해 쉬움, 구현 간단 | 채널별 세밀한 제어 불가 |
| **ACL** | 사용자별 세밀한 제어 | 관리 복잡도 폭발 |
| **ABAC** | 조건식으로 유연한 정책 | 구현·디버깅 복잡 |
| **RBAC + Channel Override** | 중간 복잡도, 대학 환경에 적합 | 커스텀 구현 필요 |

### 시행착오: 3단계로 진화

**1차 — 하드코딩 (9월)**

```kotlin
// 처음의 접근
if (role == "ADMIN") { grantAllPermissions() }
if (role == "MEMBER") { grantReadPermissions() }
```

역할이 2개뿐이었고, 모든 채널에서 같은 권한이었다. 기능이 적을 때는 이걸로 됐다.

**2차 — 엔티티 기반 RBAC (9월 말)**

역할과 권한을 엔티티로 분리했다. `GroupRole`, `GroupPermission`을 만들어서 DB에서 관리하기 시작했다. 이건 Spring + JPA로 하던 거라 자연스러웠다. 이 결정이 나중에 큰 도움이 됐다.

**3차 — RBAC + Channel Override (10월 초)**

"공지사항은 읽기만, 자유게시판은 쓰기까지" — 이 요구사항이 2차 모델로는 불가능했다. `ChannelRoleBinding` 엔티티를 추가해서, 역할이 채널별로 다른 권한을 가질 수 있게 했다.

핵심 원칙은 **Secure by Default**다. 새 채널은 바인딩 0개로 시작해서, 아무도 접근할 수 없다. 그룹장이 명시적으로 권한을 열어줘야 접근된다. 시스템 역할 3종(그룹장/교수/멤버)은 삭제 불가(불변)로 만들었다.

### N+1 쿼리 문제

권한 확인은 모든 API 요청에서 일어나니까 성능이 생명이다.

단순하게 구현하면 사용자 → 멤버십 → 역할 → 채널 10개를 순차 조회하게 돼서 13번 DB를 찌른다. Fetch Join으로 한 번에 로드하고, Caffeine 캐시를 추가해서 최종적으로 항상 2번 이하로 줄였다. 캐시 적중 시 0번.

이 부분은 JPA 성능 최적화라 Spring 하면서 익숙한 영역이었다. 오히려 나중에 프론트엔드에서 이 권한 정보를 받아서 "이 채널 보여줄지 말지" 판단하는 로직이 더 헷갈렸다.

## 6. 상태 관리: Riverpod과의 사투

### 선택

flutter_riverpod 2.6 + AsyncNotifier + Family 패턴

### 대안

| 선택지 | 장점 | 단점 |
|--------|------|------|
| **Provider (기본)** | 간단, 공식 지원 | 유연성 한계 |
| **BLoC** | 이벤트 기반, 대규모에 적합 | 보일러플레이트 많음 |
| **Riverpod** | 타입 안전, 테스트 용이 | 학습 곡선 |
| **GetX** | 간편, 올인원 | 대규모 시 혼란 |

### 왜 이게 가장 어려웠는가

Spring에서는 "상태 관리"라는 개념 자체가 거의 없다. 서버는 stateless니까, 요청이 오면 처리하고 응답 보내면 끝이다. 상태는 DB에 저장되어 있고, 필요하면 조회하면 된다.

Flutter에서는 달랐다. "지금 어떤 그룹이 선택되어 있는지", "채널 목록을 불러오는 중인지", "로딩 에러가 났는지" — 이런 걸 전부 코드에서 관리해야 한다. 이걸 "상태 관리"라고 부른다는 것도 Flutter를 하면서 처음 알았다.

Spring에서는 `@Service` 클래스에 비즈니스 로직을 넣으면 끝이다. 데이터는 Repository에서 가져오고, 결과는 DTO로 반환한다. 이 흐름이 당연했다.

Flutter에서는 이 "비즈니스 로직"을 어디에 넣어야 하는지부터 모르겠었다. 위젯 안에? Provider 안에? 별도 클래스에? 처음에는 위젯 안에 API 호출이랑 상태 변경을 다 때려넣었는데, 코드가 금방 엉망이 됐다.

### Riverpod을 고른 이유

BLoC는 Event 클래스, State 클래스를 별도로 만들어야 해서 보일러플레이트가 너무 많았다. Spring에서 Kotlin 쓰면서 보일러플레이트 줄인 게 좋았는데, 프론트엔드에서 또 늘어나는 건 싫었다.

Riverpod의 `AsyncNotifier`가 결정적이었다. 비동기 상태(로딩/에러/데이터)를 깔끔하게 관리할 수 있었다. Spring의 `@Service`처럼 "여기에 로직을 넣으면 된다"는 느낌이 들었다.

### 시행착오: Provider 순환 참조

인증 상태 변경 → 그룹 목록 갱신 → 선택 그룹 변경 → 워크스페이스 갱신이 연쇄적으로 일어나면서, 로그아웃 시 순환 참조 버그가 터졌다. A가 B를 참조하고 B가 C를 참조하는데, C가 다시 A를 참조하는 구조.

Spring에서 순환 참조가 발생하면 `@Lazy`로 해결하거나 구조를 바꾸면 된다. 근데 Riverpod에서는 Provider 간 의존성을 직접 관리해야 했다.

결국 의존성을 **단방향으로 강제**했다:

```
auth → user → groups → selectedGroup → workspace → channels → selectedChannel → posts
```

이 방향을 거스르는 참조는 전부 금지하고, 인증 로직을 별도 `AuthRepository`로 분리해서 순환을 끊었다. Spring의 레이어 분리가 여기서도 통한다는 걸 깨달았다.

## 7. 네비게이션: 라우팅이 이렇게 복잡한 거였나

### 선택

GoRouter 14.6 (Navigator 2.0 기반)

### 대안

| 선택지 | 장점 | 단점 |
|--------|------|------|
| **Navigator 1.0 (push/pop)** | 간단, 직관적 | URL 기반 라우팅 불가 |
| **auto_route** | 코드 생성으로 타입 안전 | 설정 복잡 |
| **go_router** | 선언적, URL 기반, 공식 추천 | 학습 곡선 |
| **beamer** | 유연, 커스텀 자유 | 업데이트 불규칙 |

### Spring 개발자의 라우팅 인식

Spring에서 라우팅은 간단하다. `@GetMapping("/groups")` 붙이면 `/groups` URL로 요청이 온다. 끝이다.

Flutter에서 라우팅이 이렇게 복잡한 건 상상도 못 했다. 처음에 Navigator 1.0(push/pop)으로 시작했는데, 이런 문제들이 터졌다:

- 그룹 A에서 그룹 B로 전환했는데, 뒤로가기 누르면 엉뚱한 화면이 나온다
- URL로 특정 채널에 직접 접근하면 중간 화면(그룹 선택, 워크스페이스)이 빠져서 뒤로가기가 안 된다
- `Future.microtask`랑 `setTimeout`을 조합해서 타이밍 문제를 해결하려 했는데, 간헐적으로 실패한다

Spring에서 "URL이 들어오면 그에 맞는 처리를 해라"는 너무나 자연스러운 개념인데, 프론트엔드에서는 "현재 상태가 URL을 결정하고, URL이 바뀌면 상태도 바뀐다"는 양방향 관계를 이해해야 했다. 이 개념이 처음에 어려웠다.

### Navigator 2.0으로 전환

7단계로 나눠서 점진적으로 바꿨고, 총 98개 테스트를 작성했다. 가장 까다로웠던 건 **Context-Aware Group Switching** — 그룹 전환 시 권한을 확인하고, 권한이 없으면 가장 최근 접근했던 다른 그룹으로 폴백하는 로직이었다.

선언적 방식으로 바꾸니까 "상태가 바뀌면 UI가 알아서 따라온다"가 됐다. 이건 오히려 Spring의 선언적인 느낌(`@GetMapping` 붙이면 알아서 매핑되는 것)과 비슷해서 이해가 됐다.

스크롤 위치는 LRU 캐시(최근 5개 채널)로 보존하고, 빠른 연속 클릭은 디바운싱으로 처리했다.

## 8. API 설계: 모든 응답을 하나의 형태로

### 선택

`ApiResponse<T>` 래퍼 + `ErrorCode` enum + Paged 응답

### 왜 만들었나

이건 백엔드 개발자의 습관이 통한 경우다. HTTP 상태 코드만으로는 "그룹이 없다"인지 "채널이 없다"인지 구분이 안 됐다. `error.code`를 넣어서 프론트엔드에서 에러 유형별로 다른 UI를 보여줄 수 있게 했다.

```json
성공: { "success": true,  "data": { ... }, "error": null, "timestamp": "..." }
실패: { "success": false, "data": null,    "error": { "code": "GROUP_NOT_FOUND", "message": "..." }, "timestamp": "..." }
```

ErrorCode도 도메인별로 분류했다:

```
COMMON_001~003: 공통 에러
AUTH_001~004:   인증 에러
PERMISSION_001: 권한 에러
CONTENT_001~004: 컨텐츠 에러
```

### 돌아보면

잘한 점:
- 프론트엔드에서 에러 처리 로직이 완전히 통일됐다
- 어떤 API를 호출하든 같은 방식으로 파싱 가능

아쉬운 점:
- `timestamp`를 모든 응답에 넣었는데, 프론트엔드에서 쓰는 곳이 거의 없다
- 에러 메시지 국제화(i18n)를 안 넣었다 — 현재 한글 고정

## 9. 캐싱: Caffeine의 선택

### 선택

백엔드: Caffeine 3.1.8 / 프론트엔드: Riverpod AutoDispose

### 왜 Caffeine인가

권한 확인이 모든 API 요청에서 발생하니까, 같은 사용자가 같은 채널에 연속으로 접근할 때마다 DB 찌르는 건 낭비다. 단일 서버 환경이라 Redis까지는 필요 없고, JVM 내부 캐시인 Caffeine이면 충분했다.

설정:
- 만료: 5분 (expireAfterWrite) — 권한 변경이 최대 5분 후 반영
- 크기: 10,000 entries
- 통계: 적중률 모니터링 활성화

Redis를 안 쓴 이유는 단순하다. 대학 서비스 규모(수백~수천 명)에서 Redis 띄우면 인프라 복잡도만 올라간다. 나중에 서버를 여러 대로 늘리면 그때 전환하면 된다.

## 10. 테스트 전략: 60/30/10 피라미드

### 선택

통합 테스트 60% + 단위 테스트 30% + E2E 테스트 10%

### 왜 통합 테스트를 가장 많이 잡았나

보통 테스트 피라미드는 단위 테스트가 가장 많다. 근데 이 프로젝트에서는 **Spring Context에서 JPA 쿼리랑 트랜잭션이 제대로 동작하는지**가 가장 중요했다.

단위 테스트로 서비스 로직을 검증해도, 실제 DB에서 Fetch Join이 작동하는지, 캐시 무효화 타이밍이 맞는지는 통합 테스트가 아니면 확인이 안 된다. Spring을 쓰면서 이걸 체감했기 때문에 통합 테스트 비율을 높게 잡았다.

### 도구

| 도구 | 선택 이유 |
|------|-----------|
| **MockK** (Kotlin) | Kotlin data class와 자연스러운 통합 |
| **JUnit 5** | Spring Boot 기본 |
| **mockito** (Dart) | Flutter 공식 추천 |
| **flutter_test** | 위젯 테스트 + 통합 테스트 내장 |

### 현재 상태

백엔드 155개 테스트 (133개 통과, 86%). 실패 22개는 Spring Context 캐시 충돌로 인한 Flaky 테스트다. 개별 실행하면 전부 통과하는데, 한꺼번에 돌리면 테스트 순서에 따라 결과가 달라진다. 이건 테스트 격리를 더 잘 했어야 한다.

프론트엔드 300개 이상 — Navigator 2.0 (98개), Comment (29개), Channel (86개) 등. 프론트엔드 테스트는 처음 써봤는데, `flutter_test`가 위젯을 직접 렌더링하고 상호작용할 수 있어서 생각보다 강력했다.

## 11. 아키텍처: Modular Monolith → Clean Architecture

### 선택

모놀리식 → 도메인 패키지 분리 → backend_new에서 Clean Architecture

### 3단계로 진화

**1단계 — 순수 모놀리식 (9~10월)**

17개 Controller가 한 패키지에, 34개 Service가 서로 직접 참조. 빠르게 기능을 쌓을 수 있었지만, 코드가 커지면서 한 곳 수정하면 다른 곳이 깨졌다.

**2단계 — 패키지 분리 시도 (10월 말)**

기존 코드에서 도메인별로 패키지를 나눠보려 했다. 근데 Service 간 순환 참조가 너무 깊어서 쉽게 안 풀렸다.

**3단계 — 새 프로젝트에서 재구성 (12월)**

리팩터링 대신 `backend_new/`에서 처음부터 6개 도메인으로 나눠 설계했다. 도메인 간 직접 참조 금지, 반드시 Service 인터페이스를 통해 통신.

마이크로서비스를 안 쓴 이유: 대학 서비스 규모에서 서비스 간 통신, 분산 트랜잭션, 배포 파이프라인은 오버엔지니어링이다. Modular Monolith는 나중에 필요한 도메인만 떼어낼 수 있어서 지금은 이게 맞다고 판단했다.

### 6개 도메인 의존성

```
User ← Group ← Permission
               ← Workspace ← Content
               ← Calendar
```

순환 참조 없이 의존 방향을 한쪽으로만 설정했다. 이건 Spring의 레이어 분리에서 배운 원칙이 그대로 적용된 거다.

## 12. 코드 생성: Freezed + json_serializable

### 선택

`freezed` 2.4 + `json_serializable` 6.9 + `build_runner`

### 왜 필요했나

Dart에서 불변 데이터 클래스를 만들려면 `==`, `hashCode`, `copyWith`, `toString`을 전부 직접 구현해야 한다. Kotlin에서는 `data class` 한 줄이면 끝나는 걸, Dart에서는 수십 줄 써야 했다.

Freezed가 이걸 코드 생성으로 해결해준다. Spring에서 Lombok이 보일러플레이트를 줄여주는 것과 비슷한 개념이라 이해가 빨랐다.

```dart
@freezed
class NavigationState with _$NavigationState {
  const factory NavigationState({
    @Default([]) List<WorkspaceRoute> stack,
    @Default(-1) int currentIndex,
    @Default(false) bool isLoading,
  }) = _NavigationState;
}
```

JSON 직렬화도 `json_serializable`로 자동 생성한다. Spring에서 Jackson이 알아서 해주는 JSON 변환을 Dart에서는 이렇게 코드 생성으로 처리한다.

### 아쉬운 점

- `build_runner`가 느리다 — 프로젝트가 커지면서 코드 생성에 10초 넘게 걸렸다
- 생성된 `.g.dart`, `.freezed.dart` 파일이 Git에 포함돼서 diff가 지저분하다
- Spring의 Lombok처럼 빌드 시 자동으로 되는 게 아니라, 따로 명령어를 실행해야 한다 (`flutter pub run build_runner build`)

## 13. 디자인 시스템: CSS도 모르는데 디자인 시스템이라니

### 두 가지 접근을 해봤다

**1차 — Top-down (기존 코드에 토큰 적용)**

기능을 먼저 만들고, 나중에 하드코딩된 스타일을 토큰으로 교체했다. `Color(0xFF1A1A2E)` → `AppColors.surface`, `EdgeInsets.all(16)` → `AppSpacing.md`. 390줄을 절감하고 컴포넌트를 추출했지만, 근본적으로 "덧붙인" 느낌을 벗어나지 못했다.

**2차 — Bottom-up (frontend_new에서 처음부터)**

`frontend_new/`에서 디자인 시스템을 먼저 만들고 그 위에 컴포넌트를 올렸다. 40개 시맨틱 컬러, 8단계 간격 스케일, 39개 컴포넌트를 만들고, 쇼케이스 페이지로 문서화했다.

비즈니스 로직까지는 연결하지 못했지만, 확실히 일관성이 더 좋았다.

### 백엔드 개발자에게 UI 일관성이란

Spring에서는 API 응답 형식(`ApiResponse<T>`)을 한 번 정하면 모든 엔드포인트가 같은 형식을 따른다. 이게 당연한 건데, UI에서는 이게 당연하지 않았다. 같은 버튼인데 페이지마다 색이 다르고, 간격도 제각각이었다.

디자인 시스템이라는 게 결국 "API 응답 형식을 통일하는 것"의 UI 버전이라는 걸 나중에야 깨달았다. 처음부터 토큰으로 시작했으면 훨씬 나았을 것이다.

## 14. 읽기 위치 추적: 3번의 재설계

### 시행착오

**1차 — 시간 기반**

"마지막 방문 시간" 기준으로 그 이후 작성된 글을 미읽음으로 표시했다. 간단하지만, 서버·클라이언트 시간 차이 때문에 부정확했다.

**2차 — Post ID 기반**

"마지막으로 읽은 게시글 ID"를 서버에 저장하고, 그보다 큰 ID를 미읽음으로 처리했다. 시간 차이 문제가 사라졌다. 이 부분은 백엔드 API 설계라 익숙한 영역이었다.

**3차 — Post ID + VisibilityDetector**

"언제 읽음으로 처리하느냐"가 문제였다. 스크롤해서 게시글이 화면에 보이는 순간 읽음 처리하도록 `VisibilityDetector`를 추가했다.

여기서 **Race Condition**이 터졌다. 화면을 벗어나서 Provider가 dispose된 후에도, 이미 예약된 비동기 콜백이 실행되면서 해제된 ref에 접근하는 에러가 났다. Spring에서는 이런 문제가 거의 없다 — 요청이 끝나면 끝이니까. 프론트엔드에서는 "화면이 살아있는 동안만" 동작해야 하는 코드가 있다는 게 새로웠다. `mounted` 체크를 추가해서 해결했다.

## 15. JPA + Kotlin: val에서 var로

### 시행착오

Kotlin의 철학은 불변성이다. 그래서 엔티티 필드를 전부 `val`로 선언했다.

문제는 JPA가 객체 필드를 직접 수정해서 변경을 감지한다는 것이다(dirty checking). `val`이면 값을 변경할 수 없어서 JPA update가 안 먹힌다.

결국 업데이트가 필요한 필드를 `var`로 바꿨다. User(15개 필드), Post(8개), Comment(4개), Workspace(3개), Channel(2개)이 영향을 받았다.

이건 Spring + Kotlin 조합에서 흔히 겪는 트레이드오프인데, 처음에는 "왜 val이 안 되지?"를 한참 디버깅했다. data class의 `copy()`로 새 객체를 만드는 방법도 있지만, JPA dirty checking이랑 안 맞는다.

## 16. 마이그레이션 전략: 한 번에 바꾸지 않기

### 선택

3단계 점진적 마이그레이션 (설계 완료, 실행 미완)

### 전략

기존 백엔드(backend/)와 신규 백엔드(backend_new/)를 동시에 운영하면서 점진적으로 전환하는 전략을 설계했다.

각 단계에서 10분 내 롤백 가능하도록 설계했다. Big Bang 전환은 위험하니까.

## 17. 낙관적 UI: 서버 응답을 기다리지 않는다고?

### 선택

댓글 작성 시 서버 응답 전에 즉시 화면에 반영

### 백엔드 개발자의 관점

Spring에서는 당연히 서버가 처리 완료한 후에 클라이언트에 응답을 보낸다. "처리가 끝나기 전에 결과를 보여준다"는 발상 자체가 낯설었다.

그런데 실제로 댓글을 쓰면 서버 응답까지 1~2초가 걸렸고, 그동안 화면이 멈춘 것처럼 보였다. 프론트엔드에서는 "사용자가 느끼는 속도"가 중요하다는 걸 여기서 배웠다.

```
1. 사용자가 댓글 작성 → 즉시 화면에 표시 (임시 ID)
2. 서버에 POST 요청 전송
3-a. 성공 → 서버 응답 데이터(실제 ID)로 교체
3-b. 실패 → 화면에서 제거 (롤백) + 에러 알림
```

### 까다로웠던 점

성공 시 "교체"가 예상보다 복잡했다. 임시 데이터에서 서버 데이터로 바꿀 때, 리스트의 같은 위치에 정확히 교체해야 화면이 깜빡이지 않는다. `AsyncNotifier`의 상태 갱신 패턴으로 이걸 한 곳에서 관리하게 만들었다.

이 패턴을 구현하면서 "프론트엔드에서는 사용자 경험이 곧 기능이다"라는 걸 느꼈다. 백엔드에서는 로직이 맞으면 되지만, 프론트엔드에서는 로직이 맞아도 사용자가 느끼기에 느리면 잘못된 거다.

## 전체 회고

### 잘한 선택 TOP 3

1. **권한을 초기에 엔티티로 설계한 것.** 나중에 Channel Override로 확장할 때 구조를 크게 안 바꿔도 됐다. Spring + JPA 경험이 여기서 빛을 발했다.
2. **Kotlin을 고른 것.** Java 대비 코드량이 절반 이하로 줄었고, AI 에이전트 협업 효율이 크게 올랐다.
3. **테스트를 꾸준히 작성한 것.** Navigator 2.0 전환(98개), Clean Architecture 검증(155개) 같은 대규모 변경에서 테스트가 안전망 역할을 확실히 했다.

### 아쉬운 선택 TOP 3

1. **디자인 시스템을 나중에 도입한 것.** 기능 먼저 쌓고 나중에 토큰 적용하려니 사후 작업이 많았다. API 응답 형식을 처음에 통일한 것처럼, UI도 처음부터 토큰 기반으로 갔어야 했다.
2. **DB 마이그레이션 도구를 안 넣은 것.** Flyway 없이 ddl-auto에 의존한 건 확실히 실수다.
3. **backend_new 통합을 못한 것.** 설계와 테스트까지 했는데 프로덕션 통합은 시간 부족으로 못 했다. 과정에서 많이 배웠지만, 완성까지 갔으면 더 좋았을 것이다.

### 다시 한다면

- 디자인 시스템부터: 토큰 → 컴포넌트 → 기능 순서
- Flyway를 프로젝트 시작과 동시에 설정
- 테스트 격리 개선: Spring Context 캐시 충돌 해결 (Flaky 테스트 0개)
- OpenAPI 스펙 먼저 정의하고 프론트·백 동시 개발
- 프론트엔드 상태 관리 구조를 초기에 잡기: 나중에 바꾸려면 엄청 힘들다
